백엔드 애플리케이션에서 동시성 버그는 테스트 환경보다 운영 환경에서 더 자주 드러납니다.
로컬에서는 요청이 한 번에 하나씩 들어오지만, 실제 서비스에서는 같은 사용자, 같은 상품, 같은 포인트 잔액을 여러 요청이 동시에 수정할 수 있습니다.
이때 코드만 보면 분명히 검증을 했는데도 재고가 음수가 되거나, 포인트가 중복 차감되거나, 같은 쿠폰이 두 번 발급되는 일이 생깁니다.

이 문제를 이해하려면 트랜잭션 격리 수준을 알아야 합니다.
트랜잭션 격리 수준은 여러 트랜잭션이 동시에 실행될 때 서로의 변경을 어느 정도까지 볼 수 있는지 정하는 규칙입니다.
백엔드 개발자에게 중요한 점은 "트랜잭션을 썼다"와 "동시성 문제가 해결됐다"가 같은 말이 아니라는 것입니다.
트랜잭션은 작업을 묶어주지만, 어떤 데이터를 읽고 어떤 충돌을 막을지는 격리 수준과 락 전략에 따라 달라집니다.

## Read Committed에서 생기는 착각

PostgreSQL의 기본 격리 수준은 `READ COMMITTED`입니다.
이 수준에서는 하나의 SQL 문장이 시작될 때 커밋된 데이터만 봅니다.
같은 트랜잭션 안이라도 두 번째 `SELECT`를 실행할 때 다른 트랜잭션이 이미 커밋한 변경이 있다면 다른 결과를 볼 수 있습니다.

예를 들어 포인트를 차감하는 흐름을 생각해보겠습니다.

```sql
BEGIN;

SELECT point
FROM users
WHERE id = 1;

-- 애플리케이션에서 point >= 1000 인지 확인

UPDATE users
SET point = point - 1000
WHERE id = 1;

COMMIT;
```

요청이 하나라면 자연스럽게 동작합니다.
하지만 동시에 두 요청이 들어오면 둘 다 차감 전 잔액을 읽고 검증을 통과할 수 있습니다.
`UPDATE users SET point = point - 1000`처럼 현재 값 기준으로 갱신하면 DB가 같은 행에 대한 쓰기 충돌을 순서대로 처리하긴 하지만, 검증 조건이 이미 오래된 읽기 결과에 기대고 있다면 비즈니스 규칙은 깨질 수 있습니다.

이런 경우에는 검증과 갱신을 한 문장으로 묶는 편이 안전합니다.

```sql
UPDATE users
SET point = point - 1000
WHERE id = 1
  AND point >= 1000;
```

그리고 영향받은 row 수가 1이면 성공, 0이면 잔액 부족으로 처리합니다.
핵심은 "읽고 판단하고 나중에 갱신"하는 시간을 줄이는 것입니다.
Read Committed에서는 각 문장마다 보는 스냅샷이 달라질 수 있으므로, 검증과 변경이 분리될수록 경쟁 조건이 들어올 틈이 커집니다.

## Repeatable Read가 항상 정답은 아니다

`REPEATABLE READ`는 트랜잭션이 시작된 뒤 같은 데이터를 반복해서 읽을 때 같은 스냅샷을 보게 해줍니다.
PostgreSQL에서는 MVCC를 기반으로 트랜잭션별 스냅샷을 제공하기 때문에, 읽기 일관성은 더 강해집니다.
그래서 분석성 조회나 여러 테이블을 같은 시점 기준으로 읽어야 하는 작업에서는 도움이 됩니다.

하지만 Repeatable Read가 모든 동시성 버그를 자동으로 막는 것은 아닙니다.
예를 들어 "현재 활성 예약이 10개 미만이면 새 예약을 만든다" 같은 규칙을 생각해보겠습니다.

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;

SELECT count(*)
FROM reservations
WHERE room_id = 10
  AND status = 'ACTIVE';

-- count가 10 미만이면 INSERT

INSERT INTO reservations(room_id, status)
VALUES (10, 'ACTIVE');

COMMIT;
```

두 트랜잭션이 같은 스냅샷에서 `count = 9`를 보고 동시에 insert하면, 각 트랜잭션 안에서는 모순이 없어 보일 수 있습니다.
하지만 커밋 후 전체 결과는 활성 예약 11개가 될 수 있습니다.
이런 문제는 단일 row의 갱신 충돌이 아니라 여러 row의 조건을 기반으로 한 불변식이 깨지는 형태입니다.

이때 선택지는 여러 가지입니다.
예약 수를 별도 카운터 row로 관리하고 그 row를 잠그거나, 유니크 제약 조건으로 비즈니스 규칙을 DB에 내리거나, `SERIALIZABLE` 격리 수준과 재시도 로직을 사용할 수 있습니다.
중요한 것은 "조회 조건이 여러 row에 걸친 규칙인가"를 먼저 파악하는 것입니다.
단순히 서비스 메서드에 `@Transactional`을 붙이는 것만으로는 이런 불변식을 보장하기 어렵습니다.

## Serializable과 재시도 비용

`SERIALIZABLE`은 동시에 실행된 트랜잭션 결과가 어떤 순서로 하나씩 실행된 것과 같도록 보장하려는 가장 강한 격리 수준입니다.
PostgreSQL은 Serializable Snapshot Isolation을 사용하며, 직렬화할 수 없는 충돌을 감지하면 트랜잭션 중 하나를 실패시킬 수 있습니다.
즉 Serializable은 충돌을 조용히 해결해주는 기능이라기보다, 위험한 실행을 감지하고 애플리케이션이 다시 시도하게 만드는 장치에 가깝습니다.

그래서 Serializable을 사용할 때는 반드시 재시도 전략을 함께 설계해야 합니다.

```text
1. 트랜잭션 시작
2. 비즈니스 로직 실행
3. serialization failure 발생
4. 짧은 backoff 후 재시도
5. 최대 재시도 횟수 초과 시 실패 응답
```

재시도는 무한 반복하면 안 됩니다.
트래픽이 몰린 상황에서 모든 요청이 즉시 재시도하면 충돌이 더 커질 수 있습니다.
최대 재시도 횟수, 지수 백오프, 사용자에게 보여줄 실패 메시지, 중복 요청을 막는 idempotency key까지 함께 고려해야 합니다.

## 락은 좁고 짧게 잡아야 한다

동시성 문제가 보이면 가장 먼저 떠올리는 방법이 락입니다.
PostgreSQL에서는 `SELECT ... FOR UPDATE`로 조회한 row에 row-level lock을 걸 수 있습니다.

```sql
BEGIN;

SELECT point
FROM users
WHERE id = 1
FOR UPDATE;

UPDATE users
SET point = point - 1000
WHERE id = 1;

COMMIT;
```

이 방식은 같은 사용자 row를 수정하는 요청을 순서대로 처리하게 만들 수 있습니다.
하지만 락은 성능 비용이 있습니다.
트랜잭션 안에서 외부 API 호출, 파일 처리, 긴 계산을 수행하면 락을 오래 잡고 다른 요청을 기다리게 만듭니다.
락을 잡는 코드는 가능한 한 DB 읽기와 쓰기만 포함하고, 네트워크 호출 같은 느린 작업은 트랜잭션 밖으로 빼는 편이 좋습니다.

락 순서도 중요합니다.
여러 테이블이나 여러 row를 잠글 때 요청마다 순서가 다르면 데드락이 발생할 수 있습니다.
예를 들어 A 요청은 사용자 row를 잠근 뒤 주문 row를 잠그고, B 요청은 주문 row를 잠근 뒤 사용자 row를 잠그면 서로가 서로를 기다릴 수 있습니다.
운영에서 데드락 로그가 보인다면 어떤 쿼리가 어떤 순서로 row를 잡는지 먼저 확인해야 합니다.

## 운영에서 확인할 것

동시성 문제를 줄이려면 코드 리뷰에서 다음 질문을 던지는 습관이 좋습니다.

- 읽은 값으로 검증한 뒤 나중에 갱신하는 흐름인가
- 검증 조건과 변경을 하나의 SQL로 합칠 수 있는가
- 여러 row의 합계, 개수, 존재 여부로 비즈니스 규칙을 판단하는가
- DB 제약 조건으로 막을 수 있는 규칙을 애플리케이션 코드에만 두고 있지는 않은가
- 락을 잡은 트랜잭션 안에서 외부 API를 호출하고 있지는 않은가
- Serializable을 쓴다면 serialization failure 재시도가 구현되어 있는가

운영 지표로는 DB connection pool 대기 시간, lock wait, deadlock 발생 횟수, 트랜잭션 평균 시간, p95/p99 응답 시간을 함께 봐야 합니다.
특히 응답 시간이 길어지는 시점과 lock wait이 증가하는 시점이 겹친다면, 애플리케이션 병목이 아니라 트랜잭션 경합일 수 있습니다.

## 정리

트랜잭션 격리 수준은 데이터베이스의 세부 설정이 아니라 비즈니스 규칙을 얼마나 안전하게 지킬지 결정하는 기준입니다.
Read Committed에서는 검증과 갱신을 최대한 가깝게 묶고, Repeatable Read에서는 여러 row 조건의 불변식이 깨질 수 있는지 봐야 합니다.
Serializable은 강력하지만 실패와 재시도를 전제로 설계해야 합니다.
락은 필요할 때 쓰되, 좁고 짧게 잡아야 합니다.

동시성 버그는 한 번 발생하면 재현이 어렵습니다.
그래서 기능을 만들 때부터 "동시에 두 요청이 들어오면 어떻게 될까"를 질문하는 습관이 가장 좋은 예방책입니다.
