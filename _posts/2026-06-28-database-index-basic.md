데이터베이스 성능 이야기를 하면 가장 먼저 나오는 단어 중 하나가 인덱스입니다.
하지만 인덱스를 "빠르게 해주는 마법" 정도로만 이해하면 오히려 불필요한 인덱스를 많이 만들 수 있습니다.
인덱스는 조회를 빠르게 만들 수 있지만, 저장과 수정 비용도 함께 늘리는 자료구조입니다.

## 인덱스가 필요한 순간

인덱스가 없으면 데이터베이스는 조건에 맞는 행을 찾기 위해 테이블을 처음부터 끝까지 훑어야 할 수 있습니다.
행이 적을 때는 큰 문제가 아니지만, 데이터가 많아지고 조건에 맞는 행이 적다면 전체 스캔은 비효율적입니다.

예를 들어 게시글 테이블에서 특정 작성자의 글을 자주 조회한다고 가정해보겠습니다.

```sql
SELECT *
FROM posts
WHERE author_id = 10
ORDER BY created_at DESC;
```

이런 요청이 자주 발생한다면 `author_id` 또는 `author_id, created_at` 조합의 인덱스를 검토할 수 있습니다.
중요한 기준은 "자주 사용되는 조회 조건인가", "전체 데이터 중 일부만 골라내는가", "정렬이나 조인 조건에 반복해서 등장하는가"입니다.

## B-tree 인덱스 이해하기

PostgreSQL에서 `CREATE INDEX`를 사용할 때 기본으로 만들어지는 인덱스 타입은 B-tree입니다.
B-tree 인덱스는 정렬 가능한 값에 대해 동등 비교와 범위 조건을 처리하는 데 자주 쓰입니다.
`=`, `<`, `<=`, `>=`, `>` 같은 비교뿐 아니라 `BETWEEN`, `IN`처럼 이와 비슷한 조건에서도 활용될 수 있습니다.

```sql
CREATE INDEX idx_posts_author_created_at
ON posts (author_id, created_at);
```

다만 인덱스가 있다고 항상 사용되는 것은 아닙니다.
데이터베이스의 쿼리 플래너는 테이블 크기, 통계 정보, 조건의 선택도, 예상 비용을 보고 인덱스를 쓸지 순차 스캔을 할지 결정합니다.
그래서 인덱스를 추가한 뒤에는 실제 실행 계획을 확인해야 합니다.

## 인덱스의 비용

인덱스는 별도의 구조를 유지합니다.
테이블에 `INSERT`, `UPDATE`, `DELETE`가 발생하면 관련 인덱스도 함께 갱신되어야 합니다.
조회 성능을 위해 만든 인덱스가 쓰기 성능과 저장 공간을 사용한다는 뜻입니다.

그래서 모든 컬럼에 인덱스를 만드는 것은 좋은 선택이 아닙니다.
거의 조회 조건에 쓰이지 않는 컬럼, 값의 종류가 너무 적어 대부분의 행을 읽게 되는 컬럼, 이미 다른 복합 인덱스로 충분히 처리되는 컬럼에는 인덱스가 기대만큼 도움이 되지 않을 수 있습니다.

## 확인하는 습관

백엔드 개발자가 인덱스를 다룰 때는 감으로만 판단하지 않는 것이 중요합니다.
느린 쿼리를 찾고, 어떤 조건이 병목인지 보고, 인덱스를 추가한 뒤 실행 계획과 응답 시간을 다시 확인해야 합니다.

```sql
EXPLAIN ANALYZE
SELECT *
FROM posts
WHERE author_id = 10
ORDER BY created_at DESC;
```

인덱스는 성능 개선의 강력한 도구지만, 데이터와 쿼리 패턴에 맞을 때만 효과가 있습니다.
조회 조건을 먼저 관찰하고, 필요한 곳에 작게 추가하고, 실제로 사용되는지 확인하는 흐름이 가장 안전합니다.

참고한 공식 문서: [PostgreSQL Indexes](https://www.postgresql.org/docs/current/indexes.html), [PostgreSQL Index Types](https://www.postgresql.org/docs/current/indexes-types.html)
