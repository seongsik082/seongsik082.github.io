REST API를 설계할 때 많은 팀이 인증, 상태 코드, 페이징까지는 신경 씁니다.
그런데 실제 운영에서 더 곤란한 문제는 "누가 마지막으로 저장했는가"보다 "중간에 다른 사람이 바꾼 사실을 어떻게 감지할 것인가"입니다.
게시글 편집, 상품 정보 수정, 관리자 콘솔의 설정 변경처럼 같은 리소스를 여러 사용자가 수정하는 화면에서는 lost update가 생각보다 자주 생깁니다.

전형적인 흐름은 이렇습니다.
사용자 A와 B가 같은 리소스를 `GET`으로 읽습니다.
A가 먼저 수정해 저장하고, B는 자신이 처음 읽은 오래된 복사본을 기준으로 다시 `PUT`을 보냅니다.
서버가 아무 조건 없이 마지막 요청을 받아들이면, A의 변경은 조용히 덮어써집니다.
이 문제는 DB의 낙관적 락만으로 끝나는 문제가 아니라, HTTP 계층에서 "내가 수정하는 대상이 내가 읽었던 바로 그 버전인가"를 확인해야 하는 문제이기도 합니다.

RFC 9110은 `If-Match` 헤더를 이런 목적에 맞게 정의합니다.
문서에 따르면 `If-Match`는 현재 표현의 엔티티 태그가 요청에 담긴 태그와 일치할 때만 메서드를 수행하게 만듭니다.
또 state-changing method에서 accidental overwrite, 즉 lost update를 막는 데 가장 자주 사용된다고 명시합니다.
조건이 거짓이면 원 서버는 메서드를 수행하면 안 되고, 보통 `412 Precondition Failed`로 응답합니다.

## 기본 흐름은 GET에서 ETag를 받고, PUT/PATCH에서 If-Match를 보낸다

가장 단순한 패턴은 다음과 같습니다.

```http
GET /articles/10 HTTP/1.1
```

```http
HTTP/1.1 200 OK
ETag: "article-10-v42"
Content-Type: application/json

{
  "id": 10,
  "title": "API 설계 메모",
  "body": "..."
}
```

클라이언트는 이 `ETag`를 기억해 두었다가 수정 시 다시 보냅니다.

```http
PUT /articles/10 HTTP/1.1
If-Match: "article-10-v42"
Content-Type: application/json

{
  "title": "API 설계 메모 v2",
  "body": "..."
}
```

이때 서버의 현재 ETag가 여전히 `"article-10-v42"`라면 수정합니다.
이미 다른 사용자가 바꿔서 `"article-10-v43"`이 됐다면, 서버는 요청을 적용하지 않고 `412 Precondition Failed`를 돌려야 합니다.
이 방식의 장점은 충돌을 "나중에 데이터가 이상해진 뒤 수습"하는 대신, 쓰기 시점에 드러낸다는 점입니다.

## ETag는 캐시만을 위한 헤더가 아니다

많은 개발자가 `ETag`를 `304 Not Modified` 캐시 최적화와만 연결합니다.
물론 그것도 중요한 용도입니다.
하지만 같은 RFC 9110에서 `If-Match`는 상태 변경 요청의 전제 조건으로도 쓰입니다.
즉 ETag는 "읽기 캐시 검증자"이면서 동시에 "쓰기 충돌 검증자"입니다.

이 구분이 중요한 이유는 API 설계 기준이 달라지기 때문입니다.
단순 캐시만 생각하면 약한 변경도 너그럽게 다루려 하기 쉽습니다.
반면 lost update 방지에서는 "다른 누군가가 이 표현을 바꿨는가"를 엄격히 감지해야 하므로, RFC도 `If-Match` 비교에 강한 비교(strong comparison)를 쓰라고 요구합니다.

## 412와 428을 어떻게 나눌까

조건부 요청을 도입할 때는 상태 코드 정책도 함께 정해야 합니다.
RFC 6585는 `428 Precondition Required`를 정의하면서, 서버가 요청을 조건부로 보내도록 요구할 수 있고 그 전형적 용도가 lost update 방지라고 설명합니다.

이 기준을 실무에 옮기면 보통 다음처럼 나눌 수 있습니다.

- `412 Precondition Failed`: `If-Match`는 보냈지만 현재 버전과 맞지 않다.
- `428 Precondition Required`: 이 리소스 수정에는 원래 `If-Match`가 필요한데, 아예 헤더 없이 수정 요청이 왔다.

예를 들면 관리자 설정, 문서 편집, 재고 조정 같은 "마지막 저장이 덮어쓰면 안 되는" 리소스는 아예 `If-Match`를 필수로 요구하는 편이 낫습니다.
이 경우 클라이언트가 헤더 없이 `PUT`을 보내면 `428`로 막고, 오래된 ETag를 보내면 `412`로 막으면 됩니다.

## 서버 구현은 버전 필드와 자연스럽게 연결된다

애플리케이션 내부에서는 보통 DB 버전 컬럼이나 수정 시각과 연결해 ETag를 만듭니다.
JPA를 쓴다면 `@Version` 필드를 그대로 활용하기 좋습니다.

```java
@Entity
public class Article {
    @Id
    private Long id;

    @Version
    private Long version;
}
```

```java
String etag = "\"article-" + article.getId() + "-v" + article.getVersion() + "\"";
```

중요한 건 ETag 생성 규칙이 "표현의 현재 상태를 안정적으로 식별"해야 한다는 점입니다.
단순히 현재 시각을 넣으면 읽기마다 값이 달라져 충돌 검증자로 쓰기 어렵고, 반대로 실제 변경이 있었는데 값이 안 바뀌면 검증 의미가 사라집니다.
그래서 버전 컬럼, row version, 내용 해시처럼 변경과 함께 갱신되는 기준을 쓰는 편이 낫습니다.

## 언제 쓰고, 언제 과한가

ETag/If-Match는 다음 상황에서 특히 유용합니다.

- 여러 사용자가 같은 리소스를 편집할 수 있는 관리자/협업 화면
- 모바일 앱이 오래된 데이터를 들고 다시 저장할 수 있는 환경
- 마지막 저장이 이전 저장을 조용히 덮어쓰면 안 되는 설정/문서/상품/권한 리소스

반대로 append-only 명령이나 중복 방지가 더 중요한 작업은 다른 장치가 더 중요할 수 있습니다.
예를 들어 결제 생성, 주문 생성은 리소스 덮어쓰기보다 중복 실행이 핵심이므로 `Idempotency-Key`가 더 직접적인 해법일 수 있습니다.
즉 `If-Match`는 "같은 리소스의 버전 충돌" 문제에 강하고, `Idempotency-Key`는 "같은 요청의 중복 실행" 문제에 강합니다.
둘은 대체 관계가 아니라 문제 영역이 다릅니다.

## 운영에서 확인할 것

조건부 요청을 도입했다면 다음을 봐야 합니다.

- `412`와 `428` 비율이 어느 API에서 집중되는지
- 특정 화면에서 stale data 저장 시도가 반복되는지
- 충돌 후 클라이언트가 최신 데이터를 다시 가져와 재편집하는 UX가 준비됐는지
- ETag 생성 규칙이 실제 변경을 빠뜨리거나 과도하게 자주 바꾸지 않는지

특히 `412`가 많다는 사실만 보고 서버 오류로 취급하면 안 됩니다.
그건 동시 수정이 실제로 발생하고 있고, 서버가 조용한 덮어쓰기를 막고 있다는 신호일 수 있습니다.
오히려 중요한 건 충돌 후 사용자에게 어떻게 재시도를 안내할지입니다.

## 정리

REST API에서 lost update를 막으려면 DB 내부 락뿐 아니라 HTTP 계층의 조건부 요청도 함께 설계하는 편이 좋습니다.
`ETag`는 단순 캐시 헤더가 아니라 현재 표현의 버전을 드러내는 검증자이고, `If-Match`는 그 버전이 맞을 때만 수정하게 만드는 장치입니다.
충돌 시 `412`, 조건 없는 수정 자체를 금지할 때 `428`을 쓰면 API 의미가 선명해집니다.
같은 리소스를 여러 사용자가 수정할 수 있는 순간, 조건부 쓰기는 선택이 아니라 안전장치에 가깝습니다.
