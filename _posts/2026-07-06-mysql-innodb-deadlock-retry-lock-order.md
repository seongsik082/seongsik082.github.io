운영 중인 주문 서비스에서 가끔 `Deadlock found when trying to get lock; try restarting transaction` 오류가 보이면 처음에는 DB 장애처럼 느껴진다. 같은 API를 다시 호출하면 대부분 성공하고, 로그를 보면 특정 시간대에만 몰려 있다. 그래서 "격리 수준을 낮추면 해결될까", "커넥션 풀이 부족한가" 같은 방향으로 먼저 의심하기 쉽다.

하지만 InnoDB 데드락은 단순히 DB가 느려졌다는 뜻이 아니다. 두 개 이상의 트랜잭션이 서로 상대가 가진 락을 기다리면서 더 이상 앞으로 나아갈 수 없는 상태다. 특히 주문 상태 변경, 재고 차감, 포인트 적립처럼 여러 행이나 여러 테이블을 한 트랜잭션에서 갱신하는 코드에서 자주 드러난다.

중요한 관점은 데드락을 "완전히 없애야 할 버그"로만 보지 않는 것이다. 데드락 가능성은 동시 쓰기 시스템에서 현실적으로 남을 수 있다. 대신 발생 빈도를 낮추고, 발생했을 때 사용자가 실패로 느끼지 않도록 재시도 가능한 트랜잭션 경계를 만드는 것이 운영적으로 더 안전하다.

## 문제 상황

예를 들어 주문 취소와 재고 보정 작업이 같은 테이블을 다른 순서로 갱신한다고 해보자.

```sql
-- 트랜잭션 A: 주문 취소
START TRANSACTION;
UPDATE orders SET status = 'CANCELED' WHERE id = 10;
UPDATE inventory SET quantity = quantity + 1 WHERE product_id = 7;
COMMIT;

-- 트랜잭션 B: 재고 보정 후 주문 상태 재계산
START TRANSACTION;
UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 7;
UPDATE orders SET status = 'CONFIRMED' WHERE id = 10;
COMMIT;
```

트랜잭션 A가 `orders`의 락을 잡고, 트랜잭션 B가 `inventory`의 락을 잡은 뒤 서로 다음 락을 기다리면 둘 다 진행할 수 없다. InnoDB는 데드락 감지가 켜져 있으면 한 트랜잭션을 희생자로 골라 롤백한다. 애플리케이션 입장에서는 한 요청이 실패한 것처럼 보이지만, DB 입장에서는 전체 시스템이 멈추지 않도록 하나를 되돌린 것이다.

이때 실패한 요청을 무조건 사용자에게 노출하면 간헐적인 500 오류가 된다. 반대로 아무 생각 없이 계속 재시도하면 같은 락 순서 충돌을 더 키울 수 있다. 재시도는 필요하지만, 짧은 횟수와 지연 시간을 둔 제한된 재시도여야 한다.

## 핵심 개념

데드락은 "기다리면 언젠가 풀리는 락 대기"와 다르다. 일반적인 락 대기는 앞 트랜잭션이 커밋하거나 롤백하면 다음 트랜잭션이 진행한다. 데드락은 서로가 서로의 완료를 기다리는 순환 구조라서 외부 개입 없이는 풀리지 않는다.

InnoDB 문서는 데드락을 줄이는 방법으로 트랜잭션을 짧게 유지하고, 여러 테이블이나 넓은 범위를 갱신할 때 같은 순서로 작업하며, `UPDATE ... WHERE`나 `SELECT ... FOR UPDATE`에 필요한 인덱스를 만들 것을 권장한다. 여기서 "같은 순서"가 실무에서 특히 중요하다. 서비스 코드마다 `주문 -> 재고`와 `재고 -> 주문` 순서가 섞이면 부하가 늘 때 데드락 빈도가 급격히 올라간다.

격리 수준을 바꾸면 모든 데드락이 사라진다고 기대하는 것도 위험하다. 격리 수준은 읽기 동작과 일관성에 큰 영향을 주지만, 데드락은 주로 쓰기 락의 순서와 범위 때문에 발생한다. 따라서 원인을 확인하지 않고 격리 수준부터 낮추면 정합성만 약해지고 문제는 남을 수 있다.

## 코드로 보기

Spring 서비스라면 데드락 예외를 처리할 때 트랜잭션 전체를 다시 실행해야 한다. 실패한 SQL 한 줄만 재실행하면 앞에서 읽은 값, 검증 결과, 상태 전이가 이미 롤백되었거나 낡은 판단이 되었을 수 있다.

```java
public void cancelOrderWithRetry(long orderId) {
    int maxAttempts = 3;

    for (int attempt = 1; attempt <= maxAttempts; attempt++) {
        try {
            orderCancelTx.cancel(orderId); // @Transactional 경계 전체
            return;
        } catch (DeadlockLoserDataAccessException e) {
            if (attempt == maxAttempts) {
                throw e;
            }
            sleepBackoff(attempt); // 50ms, 100ms처럼 짧고 제한된 지연
        }
    }
}
```

여기서 `orderCancelTx.cancel()`은 같은 클래스의 자기 호출이 아니라 별도 빈을 통해 호출되어야 실제 트랜잭션 프록시가 적용된다. 또한 외부 결제 취소, 알림 발송 같은 부작용은 트랜잭션 재시도 안에 직접 넣지 않는 편이 안전하다. 재시도 중 같은 알림이 여러 번 나가면 데드락보다 더 눈에 띄는 장애가 된다.

## 자주 하는 실수

첫 번째 실수는 데드락을 모두 DB 설정으로 해결하려는 것이다. `innodb_lock_wait_timeout` 값을 늘리면 일반 락 대기는 더 오래 기다릴 수 있지만, 데드락 구조 자체가 사라지는 것은 아니다. 데드락 감지가 꺼진 환경에서는 타임아웃에 의존하게 되므로 장애 체감 시간이 더 길어질 수도 있다.

두 번째 실수는 트랜잭션 안에서 너무 많은 일을 하는 것이다. 사용자 검증, 외부 API 호출, 긴 계산, 파일 처리까지 같은 트랜잭션에 넣으면 락을 잡고 있는 시간이 늘어난다. 락 보유 시간이 길수록 다른 요청이 끼어들 기회가 많아지고, 데드락과 락 대기 모두 늘어난다.

세 번째 실수는 인덱스 없이 넓은 범위를 갱신하는 것이다. `UPDATE inventory SET ... WHERE product_id = ?`에 적절한 인덱스가 없으면 필요한 행 하나를 바꾸려는 의도였더라도 더 많은 레코드나 범위를 훑으며 락을 잡을 수 있다. 데드락 분석은 애플리케이션 코드 순서와 실행 계획을 함께 봐야 한다.

## 언제 어떻게 줄일까

데드락이 한 달에 한두 번 발생하고 재시도로 사용자 영향이 없다면, 먼저 로깅과 재시도 정책을 정리하는 정도로 충분할 수 있다. 반대로 특정 API에서 매일 반복되고, 같은 테이블 조합이 계속 보인다면 트랜잭션 구조를 바꿔야 한다.

가장 먼저 정할 기준은 "락을 잡는 순서"다. 예를 들어 주문 도메인에서는 항상 `orders -> order_items -> inventory -> points` 순서로 갱신한다고 팀 규칙을 만든다. 배치, 관리자 기능, 보상 트랜잭션도 같은 순서를 따라야 한다. 운영 장애는 보통 일반 사용자 API가 아니라 예외 처리용 관리자 기능에서 같은 규칙을 깨며 발생한다.

두 번째 기준은 "재시도해도 안전한가"다. 같은 트랜잭션을 다시 실행했을 때 결과가 한 번 실행한 것과 같아야 한다. 이를 위해 상태 전이는 현재 상태 조건을 포함해 갱신하고, 외부 부작용은 outbox나 커밋 후 처리로 분리하는 편이 좋다.

```sql
UPDATE orders
SET status = 'CANCELED'
WHERE id = ?
  AND status IN ('PAID', 'CONFIRMED');
```

이런 조건부 갱신은 재시도 중 이미 다른 요청이 상태를 바꾼 경우를 감지하게 해준다. 단순히 `id`만 보고 덮어쓰는 것보다 장애 후 복구가 쉽다.

## 운영에서 볼 것

MySQL에서는 `SHOW ENGINE INNODB STATUS`로 마지막 InnoDB 데드락 정보를 확인할 수 있다. 자주 발생한다면 `innodb_print_all_deadlocks`로 모든 데드락을 error log에 남기는 것도 검토할 수 있다. 다만 로그가 늘어날 수 있으므로 운영 환경에서는 기간을 정해 켜고, 분석 후 끄는 방식이 좋다.

애플리케이션에서는 데드락 예외 수, 재시도 성공률, 최종 실패 수, 트랜잭션 실행 시간을 같이 본다. 데드락 수만 보면 "재시도했으니 괜찮다"고 착각할 수 있다. 재시도 때문에 p95, p99 응답 시간이 늘고 있다면 사용자 경험에는 이미 영향이 있다.

DB 관점에서는 느린 쿼리 로그와 실행 계획도 확인한다. 데드락 로그에 나온 SQL이 넓은 범위를 스캔한다면 락 순서보다 인덱스 설계가 먼저일 수 있다. 반대로 실행 계획은 좋아도 두 트랜잭션의 갱신 순서가 반대라면 코드 구조가 핵심이다.

## 정리

InnoDB 데드락은 동시 쓰기 서비스에서 완전히 제거하기보다 빈도와 영향을 줄이는 방식으로 다뤄야 한다. 트랜잭션은 짧게 유지하고, 여러 리소스를 갱신할 때는 팀 전체가 같은 락 순서를 따라야 한다. 데드락이 발생하면 트랜잭션 전체를 제한적으로 재시도하되, 외부 부작용이 중복되지 않게 분리해야 한다. 운영에서는 데드락 로그, 재시도 성공률, 응답 지연을 함께 보고 구조 수정이 필요한지 판단하자.
