분산 시스템에서 가장 까다로운 실패 중 하나는 "처리가 끝났는지, 응답만 잃어버린 것인지" 구분되지 않는 상황입니다.
클라이언트는 타임아웃을 보고 실패라고 판단하지만, 서버는 이미 주문 생성이나 결제 승인까지 끝냈을 수 있습니다.
이 상태에서 무작정 재시도하면 사용자는 같은 버튼을 한 번 눌렀는데 주문이 두 번 생성되거나 포인트가 두 번 차감되는 사고를 겪습니다.

실무에서는 네트워크 단절, 프록시 타임아웃, 로드밸런서 재시도, 모바일 앱의 자동 재전송처럼 "같은 의도지만 여러 번 도착하는 요청"을 피할 수 없습니다.
그래서 신뢰성 있는 API는 재시도를 막는 대신, 재시도가 와도 같은 비즈니스 작업으로 해석되게 만들어야 합니다.
이때 가장 널리 쓰는 장치가 `idempotency key`입니다.

## HTTP의 멱등성과 비즈니스 멱등성은 다르다

RFC 9110은 `PUT`, `DELETE` 같은 메서드를 멱등적(idempotent)이라고 설명합니다.
같은 요청을 여러 번 보내도 의도한 서버 상태가 한 번 보낸 것과 같아야 한다는 뜻입니다.
반대로 `POST`는 일반적으로 멱등하지 않습니다.

문제는 실무의 중요한 쓰기 작업이 대부분 `POST`라는 점입니다.
주문 생성, 결제 승인, 쿠폰 발급, 메시지 발행은 새 작업을 만드는 성격이 강하므로 같은 본문을 두 번 보내면 두 개의 결과가 생길 수 있습니다.
따라서 "HTTP 메서드가 멱등한가"와 "비즈니스 작업이 중복 실행되지 않는가"는 별개의 설계 문제입니다.

예를 들어 결제 API가 다음처럼 생겼다고 가정해보겠습니다.

```http
POST /payments
Idempotency-Key: pay-20260629-user123-order889
Content-Type: application/json

{
  "orderId": 889,
  "amount": 12900,
  "currency": "KRW"
}
```

서버는 `Idempotency-Key`를 기준으로 "이 요청이 처음인지, 이미 처리된 작업의 재시도인지"를 판단합니다.
같은 키와 같은 의미의 요청이 다시 오면 결제를 새로 실행하지 않고 첫 결과를 재사용해야 합니다.

## 안전한 구현의 핵심은 키 저장과 비즈니스 변경의 원자성이다

Idempotency key를 헤더로 받기만 하면 해결되는 것은 아닙니다.
가장 흔한 실수는 키를 따로 저장하고 비즈니스 작업을 따로 커밋하는 것입니다.
이 경우 서버가 결제를 승인한 뒤 키 저장 전에 죽으면, 다음 재시도는 "처음 보는 키"로 오인되어 같은 작업을 또 실행할 수 있습니다.

안전한 구현은 보통 다음 순서를 따릅니다.

1. `idempotency_key`와 요청의 의미를 식별할 필드(orderId, operation type 등)를 같은 트랜잭션 범위에서 검사한다.
2. 처음 들어온 키면 처리 중 상태를 기록한다.
3. 실제 비즈니스 변경과 외부 호출 결과를 저장한다.
4. 성공 또는 실패 결과를 같은 키에 연결해 재시도 시 재사용한다.

예를 들어 관계형 데이터베이스를 쓴다면 유니크 제약 조건으로 중복 키를 막을 수 있습니다.

```sql
CREATE TABLE api_idempotency (
    idempotency_key varchar(100) PRIMARY KEY,
    request_hash varchar(64) NOT NULL,
    status varchar(20) NOT NULL,
    response_code integer,
    response_body jsonb,
    created_at timestamptz NOT NULL
);
```

그리고 애플리케이션은 대략 이런 흐름을 가집니다.

```text
1. 키 조회
2. 없으면 processing 상태로 insert
3. 주문/결제 트랜잭션 실행
4. 결과 저장 후 committed
5. 동일 키 재요청 시 저장된 결과 반환
```

여기서 중요한 판단 기준은 "동일 키인데 본문이 다른 경우"입니다.
같은 키로 금액이 다른 결제 요청이 들어오면 재시도로 취급하면 안 됩니다.
실무에서는 요청 본문의 핵심 필드를 정규화해 해시로 저장하고, 동일 키에 다른 의미의 요청이 오면 `409 Conflict`나 명시적 에러로 차단하는 편이 안전합니다.

## 재시도는 반드시 필요하지만, 무제한 재시도는 더 큰 장애를 만든다

AWS Builders' Library는 분산 시스템에서 재시도가 기본 복구 수단이라고 설명합니다.
일시적 네트워크 오류나 순간적인 과부하는 재시도 한두 번으로 복구되는 경우가 많습니다.
하지만 멱등성 보장 없이 재시도를 켜면 장애 복구 장치가 곧바로 중복 실행 장치가 됩니다.

그래서 클라이언트와 서버는 함께 설계되어야 합니다.

- 클라이언트는 타임아웃과 재시도 횟수를 제한한다.
- 서버는 같은 키에 대해 같은 결과를 반환한다.
- 백오프와 지터를 넣어 몰린 재시도가 한꺼번에 다시 치지 않게 한다.
- 외부 결제사나 메시지 브로커처럼 비가역 부작용이 있는 단계는 별도 추적 키를 남긴다.

특히 "응답 코드가 500이면 무조건 새로 시도"는 위험합니다.
서버 내부 오류로 보였더라도 실제 부작용은 이미 발생했을 수 있기 때문입니다.
이 경우에는 재시도 전에 해당 idempotency key의 최종 상태를 조회할 수 있는 API를 두거나, 같은 키로 재요청 시 서버가 이전 처리 결과를 결정적으로 돌려주는 편이 낫습니다.

## outbox 없이 외부 부작용을 직접 호출하면 빈틈이 생긴다

주문을 DB에 저장한 뒤 곧바로 메시지 브로커로 "주문 생성 이벤트"를 발행하는 코드는 흔합니다.
문제는 DB 커밋 성공과 메시지 발행 성공 사이에 프로세스가 죽을 수 있다는 점입니다.
이 경우 주문은 생겼는데 다운스트림 시스템은 이벤트를 못 받아 후속 처리가 멈춥니다.

idempotency key를 도입하더라도 이 간격은 자동으로 사라지지 않습니다.
외부 부작용이 중요하다면 outbox pattern처럼 "먼저 로컬 트랜잭션에 이벤트를 기록하고, 별도 발행기가 재시도 가능하게 전송"하는 구성이 더 안전합니다.
즉, idempotency는 중복 실행을 줄이는 장치이고, outbox는 상태 저장과 외부 전파 사이의 손실 구간을 줄이는 장치입니다.
둘은 경쟁 관계가 아니라 함께 써야 할 때가 많습니다.

## 운영에서 무엇을 봐야 하나

운영 지표는 단순 성공률보다 "재시도와 중복 방지 로직이 실제로 얼마나 작동했는가"를 보여줘야 합니다.

- 동일 `Idempotency-Key` 재사용 횟수
- 키 충돌 중 본문 불일치 비율
- 타임아웃 후 최종 성공한 요청 수
- 외부 결제/주문 API의 재시도 횟수와 p95 지연 시간
- 중복 주문 차단 건수, 중복 결제 의심 로그

로그에도 `idempotency_key`, `order_id`, `payment_id`, 최종 상태를 함께 남기는 편이 좋습니다.
그래야 사용자가 "결제가 두 번 된 것 같다"고 문의했을 때 단일 요청 흐름이 아니라 여러 재시도와 외부 연동 기록을 한 번에 추적할 수 있습니다.

## 언제 적용하고, 언제 과한가

다음 조건이면 idempotency key를 거의 기본값으로 보는 편이 맞습니다.

- 사용자가 같은 의도를 다시 누를 가능성이 큰 작업
- 네트워크 타임아웃 후 재시도가 자동으로 발생하는 모바일/프록시 환경
- 결제, 주문, 포인트 적립처럼 중복 비용이 큰 작업
- 외부 시스템 연동으로 성공 여부가 즉시 확정되지 않는 작업

반대로 조회 API나 중복 실행 비용이 거의 없는 내부 관리성 작업이라면 굳이 복잡한 키 저장소를 두지 않아도 됩니다.
다만 "중복 실행 비용이 작다"는 판단은 실제 장애 비용을 기준으로 해야 합니다.
사용자 보상, 운영자 수동 정정, 회계 불일치 비용이 크다면 구현 복잡도보다 멱등성 보장이 훨씬 싸게 먹힙니다.

## 정리

재시도는 장애를 줄이는 기본 도구지만, 멱등성 없이 켜면 중복 작업을 양산합니다.
`POST` 요청은 기본적으로 멱등하지 않으므로 비즈니스 단위의 `idempotency key`를 따로 설계해야 합니다.
핵심은 키 기록과 실제 상태 변경을 최대한 원자적으로 묶고, 동일 키의 재요청에는 같은 결과를 되돌려주는 것입니다.
결제, 주문, 포인트 같은 비가역 작업일수록 이 설계를 미루지 않는 편이 안전합니다.
