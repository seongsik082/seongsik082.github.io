Redis는 백엔드에서 자주 사용하는 인메모리 데이터 저장소입니다.
DB보다 빠르게 데이터를 읽고 쓸 수 있기 때문에 캐시, 세션 저장소, 랭킹, 분산 락 같은 용도로 많이 사용됩니다.
하지만 Redis를 붙인다고 항상 서비스가 좋아지는 것은 아닙니다.

## 캐시 대상부터 정하기

캐시는 모든 데이터에 적용하는 기능이 아닙니다.
먼저 "자주 읽히고", "상대적으로 덜 바뀌며", "조회 비용이 큰 데이터"를 찾는 것이 좋습니다.

예를 들면 다음과 같은 데이터가 후보가 될 수 있습니다.

- 메인 화면에 반복해서 노출되는 인기 게시글
- 매번 조인이 많이 필요한 통계 데이터
- 외부 API 호출 결과
- 로그인 사용자 세션 정보

반대로 자주 바뀌고 정확성이 매우 중요한 데이터는 캐시 적용 전에 더 조심해야 합니다.

## 만료 시간은 왜 중요할까

캐시에는 보통 TTL을 둡니다.
TTL은 데이터가 캐시에 머무를 수 있는 시간입니다.

```text
key: post:popular
ttl: 60 seconds
```

TTL이 너무 짧으면 캐시 효과가 작고, 너무 길면 오래된 데이터를 보여줄 수 있습니다.
그래서 데이터 성격에 따라 만료 시간을 다르게 잡는 것이 좋습니다.

## 캐시 무효화

캐시에서 가장 어려운 부분은 저장보다 삭제입니다.
DB 데이터가 바뀌었는데 캐시가 그대로 남아 있으면 사용자는 이전 데이터를 보게 됩니다.

대표적인 방법은 두 가지입니다.

1. 데이터 변경 시 관련 캐시를 삭제한다.
2. 짧은 TTL을 두고 자연스럽게 만료되게 한다.

중요한 데이터라면 변경 시점에 캐시를 명시적으로 지우는 편이 안전합니다.
조회 성능이 더 중요하고 약간의 지연 허용이 가능하다면 TTL 기반 전략도 괜찮습니다.

## 백엔드 개발자가 기억할 점

Redis는 성능을 높이는 좋은 도구지만, 데이터 흐름을 더 복잡하게 만듭니다.
캐시 적용 전에는 다음 질문을 먼저 해보는 것이 좋습니다.

- 이 데이터는 정말 자주 읽히는가
- DB 조회 비용이 높은가
- 오래된 데이터를 보여줘도 되는가
- 캐시를 언제 지울 것인가
- Redis 장애 시 서비스는 어떻게 동작해야 하는가

캐시는 성능 최적화이면서 동시에 운영 설계입니다.
작게 적용하고, 효과를 측정하고, 문제가 생겼을 때 되돌릴 수 있게 만드는 것이 중요합니다.
