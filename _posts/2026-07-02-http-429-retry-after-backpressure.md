## 문제 상황

트래픽이 몰릴 때 많은 서비스가 일단 `429 Too Many Requests`를 내보냅니다. 겉으로는 맞는 대응처럼 보입니다. 그런데 실제 장애에서는 429 자체보다 그다음이 더 문제입니다. 클라이언트가 모두 1초 뒤 다시 몰려오거나, 프록시와 SDK가 제각각 재시도하면서 원래보다 더 큰 부하가 생기기 쉽기 때문입니다.

운영에서 자주 보는 패턴은 이렇습니다. API 서버는 rate limiting을 걸었다고 생각하지만, 응답에는 어떤 기준으로 막았는지 정보가 없고 `Retry-After`도 빠져 있습니다. 그러면 각 클라이언트는 자기 방식대로 즉시 재시도하거나 고정 간격으로 다시 때립니다. 결과적으로 서버는 보호된 것이 아니라 "짧은 휴식 뒤 더 큰 스파이크"를 맞습니다.

중요한 질문은 "429를 보냈는가"가 아니라 "429가 호출자 행동을 실제로 늦추는가"입니다. 상태 코드 하나만 던져서는 backpressure가 만들어지지 않습니다. 서버는 기다릴 시간을 알려 주고, 클라이언트는 그 신호를 뭉쳐서 다시 폭주하지 않도록 해석해야 합니다.

## 핵심 개념

RFC 6585는 429가 일정 시간 동안 너무 많은 요청을 보냈다는 뜻이라고 정의합니다. 또 응답 본문에 이유를 설명하는 정보를 포함하는 것이 좋고, `Retry-After` 헤더를 넣어 얼마나 기다려야 하는지 알려 줄 수 있다고 설명합니다. 그리고 429 응답은 캐시에 저장되면 안 된다고 명시합니다.

RFC 9110은 `Retry-After` 헤더의 값이 HTTP-date 또는 초 단위 숫자일 수 있다고 정의합니다. 흔히 `Retry-After: 120`처럼 초 단위로만 생각하지만, 특정 시각을 직접 보낼 수도 있습니다. 클라이언트가 이 형식을 제대로 파싱하지 못하면 의도한 대기 시간이 무시됩니다.

또 429와 503은 의미가 다릅니다. 429는 보통 호출자별 또는 키별 제한처럼 "너무 많이 보냈다"는 신호이고, 503은 서비스 자체가 잠시 불가능하다는 신호에 더 가깝습니다. 둘 다 `Retry-After`를 붙일 수 있지만, 문제의 원인을 구분해 줘야 클라이언트와 운영자가 맞는 대응을 할 수 있습니다.

## HTTP로 보기

서버는 최소한 호출자가 무엇을 해야 하는지 알 수 있는 응답을 주는 편이 좋습니다.

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
Retry-After: 60
Cache-Control: no-store

{
  "code": "RATE_LIMITED",
  "message": "Write API limit exceeded",
  "limit_scope": "user",
  "retry_after_seconds": 60
}
```

이 응답은 세 가지를 분명히 합니다.

- 지금 문제는 인증 실패가 아니라 호출량 초과다.
- 다시 시도해도 되는 시점이 대략 60초 뒤다.
- 어떤 차원에서 제한했는지(`user`, `client`, `ip`)를 추적할 수 있다.

클라이언트는 여기서 고정 재시도보다 지터가 섞인 backoff를 택하는 편이 안전합니다.

```java
Duration baseDelay = retryAfterHeader.orElse(Duration.ofSeconds(60));
Duration jitter = Duration.ofMillis(ThreadLocalRandom.current().nextLong(0, 5000));
Duration nextDelay = baseDelay.plus(jitter);

scheduler.schedule(this::retry, nextDelay);
```

`Retry-After`가 모두에게 같은 값을 주더라도, 지터가 없으면 60초 뒤 다시 같은 폭주가 생길 수 있습니다. 따라서 서버와 클라이언트는 함께 설계돼야 합니다.

## 자주 하는 실수

첫 번째 실수는 429만 주고 대기 시간을 주지 않는 것입니다. 이 경우 SDK, 브라우저, 배치 작업이 각자 다르게 행동합니다. 결과가 예측되지 않고, 잘못하면 더 공격적인 재시도를 부릅니다.

두 번째 실수는 모든 오류를 429로 뭉개는 것입니다. DB 장애나 upstream timeout으로 전체 서비스가 흔들리는 상황은 503이 더 맞는 경우가 많습니다. 429는 보통 "호출량 정책 위반"을 뜻해야 운영 지표가 섞이지 않습니다.

세 번째 실수는 재시도 가능하지 않은 요청에도 자동 재시도를 붙이는 것입니다. 결제 생성, 주문 생성처럼 부작용이 있는 쓰기 요청은 idempotency 장치 없이 반복 호출하면 다른 사고를 만듭니다. 429 처리와 idempotency 설계는 따로 생각하면 안 됩니다.

## 언제 쓰면 좋은가

429는 다음 상황에서 특히 유용합니다.

- 사용자별, 토큰별, 클라이언트별 요청량 상한을 두는 API
- 비싼 검색, 생성형 모델 호출, 결제 시도처럼 특정 연산을 보호해야 하는 API
- 악성 호출과 정상 burst를 구분하면서 부하를 천천히 낮추고 싶은 경우

반대로 서비스 전체가 유지보수 중이거나, 저장소 장애 때문에 정상 호출 자체가 불가능한 상태라면 503과 더 직접적인 장애 메시지가 맞을 수 있습니다.

실무 판단 기준을 하나만 고르면 이렇습니다. "지금 줄이고 싶은 것은 특정 호출자의 속도인가, 아니면 서비스 전체 가용성 문제인가?" 전자면 429, 후자면 503 쪽이 의미가 더 정확합니다.

## 운영에서 볼 것

- 429 응답 수와 성공 응답 대비 비율
- `Retry-After` 이후 재시도 성공률
- 동일 사용자 또는 client_id의 burst 패턴
- rate limiter 큐 길이, 토큰 버킷 잔량, upstream saturation 지표
- 429 이후 1분 안에 다시 들어온 요청 수

로그와 메트릭에서는 `limit_scope`, `client_id`, `route`, `retry_after_seconds`, `idempotency_key` 유무를 함께 보는 편이 좋습니다. 그래야 "정책이 너무 엄격한가"와 "호출자가 신호를 무시하는가"를 구분할 수 있습니다.

## 정리

429는 단순한 거절 코드가 아니라 호출 속도를 늦추기 위한 제어 신호입니다. 서버는 `Retry-After`와 제한 이유를 함께 보내고, 클라이언트는 지터가 있는 backoff로 해석해야 합니다. 그렇지 않으면 429는 부하 제어가 아니라 "잠깐 쉰 뒤 다시 몰려오는 신호"가 됩니다. 상태 코드, 헤더, 재시도 정책을 함께 설계해야 진짜 backpressure가 만들어집니다.
