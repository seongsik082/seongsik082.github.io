## 문제 상황

주문 조회 API의 스레드 풀을 가상 스레드 기반으로 바꾼 뒤 동시 접속 수가 늘었다. 기존에는 플랫폼 스레드가 부족해 요청이 대기했는데, 변경 후에는 요청을 훨씬 많이 받아들인다. 그런데 몇 분 뒤부터 DB 응답 시간이 튀고, 애플리케이션 로그에는 커넥션을 빌리는 데 오래 걸렸다는 경고가 쌓인다.

개발자는 “스레드가 충분해졌는데 왜 느려졌지?”라고 생각하기 쉽다. 가상 스레드는 요청을 담을 수 있는 그릇을 가볍게 만들어 주지만, DB 커넥션 수나 외부 결제사의 허용 동시 호출 수까지 늘려 주지는 않는다. 오히려 애플리케이션이 더 많은 요청을 한꺼번에 받아 downstream의 유한한 자원을 빠르게 소진할 수 있다.

가상 스레드의 도입 여부와 자원 확장 여부는 같은 문제가 아니다. 먼저 요청을 수용하는 스레드의 한도와 DB·외부 API·CPU 같은 실제 병목의 한도를 분리해서 봐야 한다.

## 핵심 개념

가상 스레드는 OS 스레드를 직접 하나씩 점유하는 플랫폼 스레드보다 훨씬 가벼운 Java 스레드다. JDK는 많은 가상 스레드를 소수의 플랫폼 스레드 위에서 실행한다. 가상 스레드가 `java.*` API의 blocking I/O를 기다리는 동안에는 실행을 잠시 멈추고 플랫폼 스레드를 다른 작업에 사용할 수 있다.

이 특성은 요청이 CPU 계산보다 네트워크·파일·DB 응답을 기다리는 시간이 긴 서버에서 유리하다. 반대로 CPU 계산 자체를 더 빠르게 만들지는 않는다. 코어가 8개인 서버에서 CPU-bound 작업을 가상 스레드 10,000개로 실행해도 계산 처리량이 10,000배가 되지 않는다.

JDK 21에서 가상 스레드는 정식 기능이 되었고, 공식 설계 문서는 가상 스레드를 풀에 넣기보다 작업마다 새로 만드는 방식을 권장한다. 풀은 비싼 플랫폼 스레드의 수를 제한하기 위해 쓰는 도구였지만, 가상 스레드는 생성 비용이 낮으므로 같은 방식으로 풀링할 이유가 적다.

여기서 중요한 구분이 있다. 가상 스레드는 “작업을 얼마나 많이 받아들일 것인가”를 해결하고, 세마포어·커넥션 풀·레이트 리미터는 “한정된 자원에 동시에 몇 개를 보낼 것인가”를 해결한다. 둘을 같은 숫자로 취급하면 운영 중에 병목이 숨겨진다.

## 코드로 보기

가상 스레드 자체는 작업당 만들 수 있다.

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (var orderId : orderIds) {
        executor.submit(() -> orderService.load(orderId));
    }
}
```

하지만 `orderService.load()`가 모두 같은 DB를 조회한다면, DB 커넥션 풀의 크기는 여전히 유한하다. 애플리케이션이 요청을 1,000개 받았어도 DB 커넥션이 20개라면 나머지 작업은 커넥션을 기다린다. 이 대기 시간이 길어지면 HTTP 요청 timeout, 재시도, 더 많은 DB 대기로 이어질 수 있다.

비용이 큰 downstream을 별도로 제한하는 가장 단순한 방법은 세마포어를 두는 것이다.

```java
final class OrderReader {
    private final Semaphore dbPermits = new Semaphore(20);
    private final OrderRepository repository;

    OrderReader(OrderRepository repository) {
        this.repository = repository;
    }

    Order find(long id) throws InterruptedException {
        dbPermits.acquire();
        try {
            return repository.findById(id).orElseThrow();
        } finally {
            dbPermits.release();
        }
    }
}
```

`20`은 가상 스레드 수가 아니라 DB에 동시에 보낼 작업의 상한이다. 실제 값은 DB 커넥션 풀 크기, 쿼리 평균 시간, 다른 API의 커넥션 사용량을 보고 정해야 한다. 풀 전체를 한 기능이 독점하지 않게 여유를 남기거나, 기능별로 더 작은 permit을 두는 편이 안전할 때도 있다.

## 자주 하는 실수

첫 번째 실수는 가상 스레드로 바꾸면서 외부 호출의 timeout을 없애는 것이다. 가상 스레드는 대기 중인 플랫폼 스레드를 돌려줄 수 있지만, 응답이 영원히 오지 않는 요청을 끝내 주지는 않는다. connect timeout, read timeout, 전체 요청 deadline은 여전히 필요하다.

두 번째 실수는 가상 스레드 수를 DB 커넥션 풀 크기로 제한하려고 다시 큰 플랫폼 스레드 풀을 만드는 것이다. 그렇게 하면 가상 스레드의 장점을 잃고, “요청 수 제한”과 “DB 동시성 제한”이 한 executor에 섞인다. 요청은 가상 스레드로 받아들이되, DB와 외부 API 앞에 각각 명시적인 한도를 두는 편이 원인을 설명하기 쉽다.

세 번째 실수는 모든 작업을 가상 스레드로 바꾸면 CPU 부하도 해결될 것이라고 생각하는 것이다. JSON 대량 변환, 암호화, 이미지 압축처럼 CPU가 주된 작업은 프로세서 코어 수와 CPU 스케줄링이 병목이다. 이런 작업은 별도의 제한된 executor나 작업 큐가 더 적합할 수 있다.

네 번째 실수는 많은 가상 스레드에 큰 객체를 `ThreadLocal`로 넣는 것이다. 가상 스레드도 thread-local을 지원하므로 기존 코드가 그대로 동작할 수 있지만, 요청 수가 늘면 각 스레드가 참조하는 컨텍스트도 많아진다. 인증 정보나 trace context처럼 필요한 값만 전달하고, 대형 버퍼·세션 객체를 오래 보관하지 않는지 확인해야 한다.

## 언제 쓰면 좋은가

가상 스레드는 각 요청이 외부 I/O를 기다리는 시간이 길고, 요청당 독립적인 작업을 수행하는 서버에 잘 맞는다. 기존의 동기식 코드 흐름을 유지하면서 동시 요청 수를 늘리고 싶을 때 특히 검토할 만하다. 코드가 비동기 callback으로 지나치게 복잡해진 서비스라면 가상 스레드가 가독성을 회복하는 데 도움을 줄 수 있다.

반대로 CPU-bound 작업, 이미 DB connection pool이나 외부 API quota가 포화된 서비스, 가상 스레드를 제대로 지원하지 않는 오래된 라이브러리가 많은 서비스에는 바로 전체 적용하지 않는 편이 좋다. 작은 트래픽의 내부 도구라면 도입 이득보다 런타임·관측 방식의 변경 비용이 더 클 수도 있다.

판단 기준은 “동시 요청을 늘렸을 때 기다림이 겹쳐 처리량이 늘어나는가”와 “늘어난 요청이 downstream을 압도하지 않는가”다. 두 번째 질문에 답할 수 없다면 먼저 자원별 동시성 한도와 timeout부터 정의해야 한다.

## 운영에서 볼 것

가상 스레드를 켠 뒤에는 애플리케이션 스레드 수만 보지 말고 다음 지표를 같이 본다.

- DB 커넥션 풀 active, idle, pending과 커넥션 획득 대기 시간
- DB 쿼리 p95·p99, lock wait, timeout 비율
- 외부 API별 동시 호출 수, 응답 지연, 429·5xx 비율
- HTTP in-flight 요청 수와 요청 deadline 초과 수
- CPU 사용률, GC pause, heap 사용량

특히 “가상 스레드 수가 많다”는 사실만으로 포화 원인을 판단하면 안 된다. DB 커넥션 대기 시간이 늘고 쿼리 지연도 함께 늘었다면 스레드를 더 만드는 대신 DB 동시성, 쿼리 비용, 트래픽 제한을 먼저 조정해야 한다.

장애를 재현할 때는 가상 스레드 도입 전후로 같은 부하를 주고, 동시 요청 수만이 아니라 downstream에 실제로 도착한 동시 작업 수를 비교한다. JDK가 제공하는 JFR과 가상 스레드용 스레드 관측 기능도 활용할 수 있지만, 애플리케이션 지표를 대신하지는 않는다. 서비스 관점에서는 “몇 개의 요청이 스레드에 매달렸는가”보다 “어디에서 얼마나 기다렸는가”가 더 중요하다.

## 정리

Java 가상 스레드는 많은 I/O 대기 요청을 가볍게 수용하는 도구이지, DB나 외부 API의 용량을 늘리는 도구가 아니다.
가상 스레드는 작업당 생성하고, 유한한 downstream 자원은 세마포어·커넥션 풀·레이트 리미터로 별도 제한하자.
CPU-bound 작업에는 가상 스레드 수를 늘리는 것보다 코어 수와 작업 큐를 기준으로 판단해야 한다.
도입 전후에는 커넥션 획득 대기, downstream 동시성, timeout, p99를 함께 비교해야 한다.
