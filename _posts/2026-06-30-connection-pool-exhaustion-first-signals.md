## 문제 상황

운영에서 "DB가 느린 것 같다"는 이야기가 나올 때, 실제 원인이 데이터베이스 자체가 아니라 커넥션 풀 고갈인 경우가 많습니다. 애플리케이션 스레드는 DB 연결을 받지 못해 대기하고, 응답 시간은 튀기 시작하고, 결국 일부 요청은 timeout으로 떨어집니다. 그런데 모니터링을 늦게 보면 이미 장애가 커진 뒤라 DB만 의심하게 됩니다.

커넥션 풀 문제는 특히 트래픽이 갑자기 늘거나, 외부 API 지연 때문에 트랜잭션이 길어질 때, 혹은 코드에서 커넥션을 오래 붙잡을 때 잘 드러납니다. 사용자는 "응답이 갑자기 느려졌다"고 느끼지만, 개발자 입장에서는 어느 시점부터 대기가 쌓였는지 먼저 봐야 합니다.

실무에서 중요한 포인트는 "풀 고갈이 발생했는가"보다 "고갈되기 직전에 어떤 신호가 먼저 보였는가"입니다. 그 신호를 빨리 보면 장애를 DB 다운으로 키우지 않고 애플리케이션 계층에서 먼저 완화할 수 있습니다.

## 핵심 개념

HikariCP 기준으로 `maximumPoolSize`는 실제 데이터베이스에 열 수 있는 최대 커넥션 수를 결정합니다. 풀에 여유가 없고 idle connection도 없다면, `getConnection()` 호출은 `connectionTimeout` 동안 대기하다가 실패합니다. 즉, 커넥션 풀이 고갈되면 첫 번째 증상은 보통 "연결 획득 대기"입니다.

여기서 헷갈리기 쉬운 부분이 있습니다. DB CPU가 높지 않아도 풀은 먼저 고갈될 수 있습니다. 예를 들어 쿼리는 빠르지만 애플리케이션이 트랜잭션을 길게 유지하거나, 외부 호출을 하는 동안 커넥션을 놓지 않으면 풀은 금방 찹니다. 그래서 "DB가 안 바쁘다"는 이유만으로 풀 문제를 배제하면 안 됩니다.

HikariCP 문서는 `connectionTimeout`, `maximumPoolSize`, `maxLifetime`, `keepaliveTime`을 핵심 설정으로 다룹니다. 특히 `maxLifetime`은 DB나 인프라가 강제로 연결을 끊는 시간보다 몇 초 짧게 두는 편을 강하게 권장합니다. 이 설정이 맞지 않으면 특정 시점에 커넥션이 한꺼번에 죽으면서 풀 재생성 비용이 몰릴 수 있습니다.

## 코드로 보기

Spring Boot에서 HikariCP를 쓴다면 최소한 아래 설정의 의미는 알고 있는 편이 좋습니다.

```yaml
spring:
  datasource:
    hikari:
      maximumPoolSize: 20
      connectionTimeout: 3000
      maxLifetime: 1700000
      keepaliveTime: 120000
      leakDetectionThreshold: 5000
```

- `maximumPoolSize`: 동시에 유지할 최대 커넥션 수
- `connectionTimeout`: 커넥션을 못 구했을 때 얼마나 기다릴지
- `maxLifetime`: 커넥션을 얼마 동안 살려둘지
- `keepaliveTime`: idle connection이 너무 오래 조용해서 끊기지 않도록 확인 주기
- `leakDetectionThreshold`: 너무 오래 반환되지 않는 커넥션을 경고로 남길 시간

장애 시 애플리케이션에서 먼저 보이는 코드는 대개 이런 형태입니다.

```java
try (Connection connection = dataSource.getConnection()) {
    // query
}
```

문제는 이 코드가 아니라, 이 커넥션을 오래 붙잡고 있는 주변 흐름입니다. 트랜잭션 안에서 외부 HTTP 호출을 하거나, 큰 응답 직렬화가 끝날 때까지 지연 로딩 쿼리가 섞이면 커넥션 체류 시간이 길어집니다.

## 자주 하는 실수

첫 번째 실수는 `maximumPoolSize`를 무작정 키우는 것입니다. HikariCP도 풀 크기는 실행 환경에 맞춰야 한다고 설명합니다. 커넥션을 많이 열면 오히려 DB 컨텍스트 스위칭과 경쟁이 늘어 성능이 떨어질 수 있습니다. 풀 고갈은 숫자가 작아서가 아니라 "커넥션을 오래 쓰는 흐름" 때문에 생기는 경우가 더 많습니다.

두 번째 실수는 `minimumIdle`을 억지로 따로 크게 잡는 것입니다. HikariCP는 spike demand 대응과 응답성을 위해 고정 크기 풀처럼 두는 편을 권장합니다. idle 수를 따로 조절하려다가 풀 동작을 복잡하게 만들고, 정작 대기 시간 문제는 못 잡는 경우가 있습니다.

세 번째 실수는 `leakDetectionThreshold`를 영구 모니터링 수단처럼 두는 것입니다. 이 값은 문제를 찾는 데는 도움이 되지만, 장기적으로는 로그를 많이 만들 수 있습니다. 특정 이슈를 좁히는 동안 켜고, 원인을 찾은 뒤 조정하는 편이 현실적입니다.

## 언제 쓰면 좋은가

풀 설정을 적극적으로 손봐야 하는 시점은 p95/p99 응답 시간이 먼저 튀고, 애플리케이션 스레드 대기가 늘고, DB 쿼리 시간은 상대적으로 안정적인 경우입니다. 이 패턴이면 DB 엔진보다 커넥션 획득 대기를 먼저 의심하는 편이 맞습니다.

반대로 단순 개발 환경이나 트래픽이 낮은 내부 도구에서는 숫자 미세 조정보다 코드 흐름을 정리하는 편이 더 큰 효과를 냅니다. 외부 호출을 트랜잭션 밖으로 빼고, 불필요하게 긴 트랜잭션을 줄이고, N+1이나 지연 쿼리로 커넥션 체류 시간이 늘지 않게 하는 쪽이 먼저입니다.

실무 판단 기준을 하나만 고르면 이렇습니다. "풀 고갈 경고가 보이면 최대 커넥션 수를 늘리기 전에, 한 요청이 커넥션을 붙잡고 있는 시간을 먼저 본다." 이 순서가 뒤집히면 문제를 숫자로 덮게 됩니다.

## 운영에서 볼 것

- `getConnection()` timeout 예외 건수
- 응답 시간 p95/p99 상승 시점
- active connection 수와 pending thread 수
- DB 쿼리 평균 시간과 애플리케이션 전체 응답 시간의 차이
- leak detection 로그 존재 여부
- 외부 API 호출 지연과 DB 대기가 같은 시점에 늘었는지 여부

가능하면 HikariCP 관련 메트릭을 아래 질문과 연결해서 봐야 합니다.

- active가 계속 `maximumPoolSize` 근처인가
- idle은 0인데 요청 수는 계속 증가하는가
- timeout 직전에 pending이 급격히 늘어나는가

이런 신호가 보이면 DB CPU보다 먼저 애플리케이션 요청 흐름과 트랜잭션 경계를 확인하는 편이 빠릅니다.

## 정리

커넥션 풀 고갈은 DB 장애처럼 보이지만, 실제로는 애플리케이션이 커넥션을 오래 붙잡는 흐름에서 시작되는 경우가 많습니다. `connectionTimeout`, `maximumPoolSize`, `maxLifetime` 설정을 이해하고, 숫자를 올리기 전에 커넥션 체류 시간과 요청 대기 신호를 먼저 보는 편이 안전합니다. 운영에서는 "DB가 느리다"보다 "커넥션을 못 받고 있다"는 신호를 먼저 잡아내는 것이 중요합니다.
