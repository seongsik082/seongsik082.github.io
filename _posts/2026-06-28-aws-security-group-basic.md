AWS에서 EC2나 RDS를 처음 다루면 애플리케이션은 정상적으로 실행 중인데 접속이 되지 않는 상황을 자주 만납니다.
이때 확인해야 하는 대표적인 지점이 보안 그룹입니다.
보안 그룹은 연결된 리소스에 어떤 트래픽을 허용할지 정하는 규칙 묶음입니다.

## 인바운드와 아웃바운드

보안 그룹 규칙은 크게 인바운드와 아웃바운드로 나뉩니다.
인바운드는 외부에서 리소스로 들어오는 트래픽이고, 아웃바운드는 리소스에서 외부로 나가는 트래픽입니다.
각 규칙은 보통 프로토콜, 포트 범위, 출발지 또는 목적지를 함께 가집니다.

예를 들어 Spring Boot 서버를 EC2에서 `8080` 포트로 실행했다면, 애플리케이션이 떠 있어도 보안 그룹 인바운드 규칙에 `8080` 포트가 열려 있지 않으면 외부에서 접근할 수 없습니다.
반대로 EC2가 외부 API를 호출하거나 패키지를 내려받아야 한다면 아웃바운드 규칙도 확인해야 합니다.

```text
HTTP  80    0.0.0.0/0
HTTPS 443   0.0.0.0/0
SSH   22    내 IP 주소
App   8080  로드밸런서 보안 그룹
```

이런 식으로 "누가", "어떤 포트로", "어디까지" 접근할 수 있는지 구체적으로 적어야 합니다.

## 보안 그룹은 상태를 기억한다

AWS 보안 그룹은 상태 저장 방식으로 동작합니다.
예를 들어 인스턴스에서 외부로 요청을 보냈다면, 그 요청에 대한 응답 트래픽은 인바운드 규칙에 별도로 열려 있지 않아도 돌아올 수 있습니다.
반대로 외부에서 먼저 들어오는 요청은 인바운드 규칙이 허용해야 합니다.

이 특징을 알면 "아웃바운드는 열려 있는데 왜 외부에서 접속이 안 될까" 같은 문제를 덜 헷갈릴 수 있습니다.
외부 사용자가 서버에 접속하는 흐름과 서버가 외부 서비스에 요청하는 흐름은 방향이 다르기 때문입니다.

## 0.0.0.0/0을 조심하기

`0.0.0.0/0`은 모든 IPv4 주소에서 접근을 허용한다는 뜻입니다.
웹 서비스의 80, 443 포트처럼 공개되어야 하는 경우에는 사용할 수 있지만, SSH 22번이나 RDP 3389번을 전체 공개하는 것은 위험합니다.
관리용 포트는 가능하면 자신의 고정 IP나 회사 VPN 대역처럼 필요한 범위로 제한해야 합니다.

데이터베이스도 마찬가지입니다.
RDS를 인터넷 전체에 열기보다 애플리케이션 서버의 보안 그룹에서만 접근하도록 묶으면 실수로 노출될 가능성을 줄일 수 있습니다.
백엔드 구성에서는 보안 그룹을 역할별로 나누는 습관이 도움이 됩니다.

- 외부 요청을 받는 로드밸런서용 보안 그룹
- 애플리케이션 서버용 보안 그룹
- DB나 Redis처럼 내부에서만 접근하는 저장소용 보안 그룹

## 장애를 볼 때의 순서

접속 실패를 만나면 애플리케이션 로그만 보는 대신 계층을 나누어 확인하는 것이 좋습니다.
서버 프로세스가 포트에 바인딩되어 있는지, 운영체제 방화벽이 막고 있지 않은지, 보안 그룹 인바운드 규칙이 맞는지, 퍼블릭 IP와 라우팅이 맞는지 차례로 좁혀야 합니다.

보안 그룹은 배포의 작은 설정처럼 보이지만 실제로는 서비스의 문을 어디까지 열지 정하는 중요한 설계입니다.
처음에는 조금 불편하더라도 필요한 포트와 필요한 출발지만 허용하는 방식으로 시작하는 것이 안전합니다.
