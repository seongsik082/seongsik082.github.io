## 문제 상황

AWS에서 로드밸런서를 붙일 때 "HTTP면 ALB, 그 외엔 NLB" 정도로만 기억하고 넘어가는 경우가 많습니다. 초기에는 충분할 수 있습니다. 하지만 서비스가 커지면 이 차이가 단순한 상품 선택이 아니라, 헤더 전달 방식, 정적 IP 필요 여부, 가용 영역 간 분산 방식, 헬스체크 운영까지 바꿉니다.

실제로 운영에서 자주 보는 문제는 이런 형태입니다. 클라이언트 방화벽이 고정 IP 허용만 가능해서 갑자기 NLB가 필요해졌는데, 기존 서비스는 `X-Forwarded-*` 헤더와 호스트 기반 라우팅에 기대고 있습니다. 반대로 정적 IP는 필요 없는데, TCP 수준에서만 분산되는 NLB를 붙였다가 애플리케이션 라우팅을 직접 구현하느라 복잡도가 커지기도 합니다.

즉, ALB와 NLB의 차이는 "더 고성능인가" 같은 단순 비교가 아닙니다. 요청을 어느 계층에서 이해하고, 연결을 어떤 기준으로 타깃에 보내며, 운영자가 어떤 제약을 감수할지의 차이입니다.

## 핵심 개념

AWS 문서를 보면 Application Load Balancer는 HTTP/1.1, HTTP/2, WebSocket을 지원하고, 요청을 받은 뒤 리스너 규칙을 평가해 어떤 타깃 그룹으로 보낼지 결정합니다. 또 `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Port` 헤더를 자동으로 추가합니다. 즉, ALB는 "HTTP 요청을 이해하는 계층"에서 동작한다고 보는 편이 맞습니다.

반면 Network Load Balancer는 OSI 4계층에서 동작합니다. TCP, UDP, TLS, QUIC 같은 프로토콜 단위로 연결을 받고, AWS 문서 기준으로 flow hash 알고리즘으로 타깃을 고릅니다. 각 TCP 연결은 생애주기 동안 하나의 타깃에 붙습니다. 또한 가용 영역마다 정적 IP를 가지며, 인터넷 공개형이면 서브넷별 Elastic IP도 연결할 수 있습니다.

이 차이 때문에 운영 결과가 달라집니다. ALB는 HTTP 헤더와 경로 기반 규칙, 여러 타깃 그룹 분기, WAF 같은 HTTP 계층 운영과 잘 맞습니다. NLB는 고정 IP, L4 처리, 프로토콜 투명성, 대량 연결 처리와 잘 맞습니다. 둘은 상하 관계가 아니라 전제가 다릅니다.

## 구성으로 보기

예를 들어 아래 요구가 있으면 ALB 쪽이 더 자연스럽습니다.

```text
/api/* 는 API 서버로
/admin/* 는 관리자 서버로
WebSocket 연결 지원
애플리케이션에서 X-Forwarded-For 로 원본 클라이언트 IP 확인
```

이 경우 ALB의 리스너 규칙과 HTTP 헤더 추가 기능을 그대로 활용할 수 있습니다. 애플리케이션은 HTTP 요청 맥락을 이미 받은 상태에서 동작합니다.

반대로 아래 요구는 NLB가 더 잘 맞습니다.

```text
파트너사 방화벽에 고정 IP 등록 필요
TCP 또는 TLS 그대로 전달 필요
가용 영역별 정적 주소 운영
매우 많은 동시 연결 처리
```

AWS 문서 기준으로 NLB는 가용 영역마다 네트워크 인터페이스와 정적 IP를 가지고, 타깃 그룹 단위로 헬스체크를 운영합니다. TCP/TLS/UDP 계열 서비스에서 "로드밸런서가 HTTP를 해석하지 않아야 한다"는 요구가 있으면 ALB보다 단순해집니다.

## 자주 하는 실수

첫 번째 실수는 정적 IP가 필요하다는 이유만으로 NLB로 바꾸고, 기존 애플리케이션이 기대하던 HTTP 계층 기능이 사라진다는 점을 놓치는 것입니다. ALB에서 당연하게 쓰던 헤더, 경로 기반 규칙, 호스트 기반 라우팅은 NLB의 기본 역할이 아닙니다. 이 차이를 애플리케이션 코드나 별도 프록시로 다시 메우기 시작하면 구조가 급격히 복잡해집니다.

두 번째 실수는 가용 영역 분산 기본값을 같게 보는 것입니다. AWS 문서상 ALB는 로드밸런서 수준에서 cross-zone load balancing이 항상 켜져 있고, NLB는 기본적으로 꺼져 있습니다. 따라서 NLB에서 특정 AZ에 타깃 수가 적거나 헬시 타깃이 비어 있으면 트래픽 분포가 예상과 달라질 수 있습니다.

세 번째 실수는 연결 단위 분산과 요청 단위 분산을 같은 것으로 보는 것입니다. ALB는 요청을 이해하고 라우팅 규칙을 적용하지만, NLB는 연결과 플로우 중심으로 동작합니다. HTTP keep-alive, 장수명 연결, WebSocket, 게임 서버처럼 연결 특성이 중요한 서비스에서는 이 차이가 직접적인 운영 결과로 이어집니다.

## 언제 쓰면 좋은가

ALB는 HTTP 계층 판단이 필요한 서비스에 잘 맞습니다. 경로/호스트 기반 라우팅, HTTP 헤더 활용, gRPC 또는 WebSocket 지원, 애플리케이션 앞단의 정책 분기가 중요하다면 ALB가 기본 선택입니다.

NLB는 L4 서비스이거나 정적 IP가 필수일 때 강합니다. 파트너 방화벽 허용 목록, 고정 주소 요구, TCP/TLS/UDP 처리, 높은 연결량, 타깃 그룹별 독립 헬스체크 운영이 필요하면 NLB를 우선 검토하는 편이 맞습니다.

실무 판단 기준을 하나만 고르면 이렇습니다. "애플리케이션이 HTTP 요청의 의미를 로드밸런서 단계에서 활용해야 하면 ALB, 주소와 프로토콜의 투명성이 더 중요하면 NLB"입니다. 상품 이름보다 요청을 어디까지 해석할 것인지가 선택 기준이 되어야 합니다.

## 운영에서 볼 것

- 로드밸런서 유형 변경 전후의 클라이언트 허용 목록 요구
- `X-Forwarded-*` 헤더 의존 여부
- 가용 영역별 타깃 수와 cross-zone 설정
- 연결 수, 장수명 연결 비율, WebSocket/gRPC 사용 여부
- 타깃 그룹 헬스체크 실패 패턴과 특정 AZ 쏠림

특히 NLB를 선택했다면 아래를 같이 점검하는 편이 좋습니다.

- 각 AZ에 충분한 헬시 타깃이 있는가
- 정적 IP 요구가 정말 로드밸런서 레벨 요구인가
- 애플리케이션이 HTTP 헤더 주입 없이도 필요한 정보를 얻을 수 있는가

## 정리

ALB와 NLB는 둘 다 트래픽을 분산하지만, 하나는 HTTP 요청을 이해하는 L7 로드밸런서이고 다른 하나는 연결과 프로토콜 중심의 L4 로드밸런서입니다. 그래서 헤더 처리, 정적 IP, cross-zone 기본값, 타깃 선택 방식이 모두 달라집니다. 운영에서는 "무엇이 더 좋으냐"보다 "서비스가 요청 의미를 써야 하느냐, 주소와 프로토콜 투명성이 더 중요하냐"로 선택하는 편이 안전합니다.
