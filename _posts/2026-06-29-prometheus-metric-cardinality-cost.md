관측성을 강화하겠다는 이유로 메트릭을 마구 추가하다 보면, 어느 날부터 Prometheus가 느려지고 저장 비용이 급격히 늘어나는 시점이 옵니다.
대개 원인은 수집 개수 그 자체보다 `cardinality`, 즉 시계열(series)의 개수가 폭발한 것입니다.
메트릭 이름이 같아도 라벨 조합이 달라지면 Prometheus 입장에서는 서로 다른 시계열로 저장합니다.

문제는 카디널리티 폭증이 조용히 시작된다는 점입니다.
초기에는 대시보드가 더 자세해진 것처럼 보이지만, 트래픽이 늘고 배포가 반복되면 메모리 사용량, 쿼리 지연, 원격 저장 비용이 함께 치솟습니다.
운영 중인 Prometheus가 불안정해지면 메트릭 기반 알림도 함께 흔들리므로, 관측성을 강화하려다 정작 관측 시스템을 약하게 만드는 역설이 생깁니다.

## 카디널리티는 "메트릭 수"가 아니라 "고유한 라벨 조합 수"다

Prometheus에서 다음 메트릭을 생각해보겠습니다.

```text
http_requests_total{method="GET", status="200", route="/orders/{id}"}
```

이 메트릭은 `method`, `status`, `route` 조합마다 별도 시계열이 생깁니다.
라벨 값이 적당히 제한되어 있다면 문제가 없습니다.
하지만 여기에 `user_id`, `session_id`, `request_id`, 원본 URL path처럼 값 종류가 무한히 늘어나는 필드를 넣으면 거의 요청마다 새로운 시계열이 생길 수 있습니다.

예를 들어 다음 계측은 전형적인 실수입니다.

```java
Counter.builder("http_requests_total")
    .tag("user_id", userId)
    .tag("path", request.getRequestURI())
    .register(registry)
    .increment();
```

`/orders/1`, `/orders/2`, `/orders/3`가 모두 다른 path로 기록되면 트래픽이 늘수록 시계열 수도 선형 이상으로 증가합니다.
Prometheus 공식 문서는 라벨 값으로 카디널리티가 큰 차원을 넣지 말라고 권장하고, 메트릭 한 개에서 대부분의 라벨은 가능한 한 10 이하의 값 범위를 유지하라고 안내합니다.

## 좋은 라벨은 집계를 돕고, 나쁜 라벨은 사건 로그를 흉내 낸다

메트릭은 로그를 대체하는 도구가 아닙니다.
메트릭은 "현재 어떤 현상이 얼마나 자주 일어나는가"를 값의 집계로 보여주는 데 강하고, 로그는 "개별 요청 하나에서 정확히 무슨 일이 있었는가"를 추적하는 데 강합니다.

그래서 사용자 단위, 주문 단위, 요청 단위 식별자는 대개 메트릭보다 로그나 트레이스에 실어야 합니다.
메트릭에 남길 라벨은 팀이 반복적으로 그룹화해서 볼 차원만 골라야 합니다.

좋은 예시는 다음과 같습니다.

```text
http_server_requests_total{
  method="POST",
  route="/orders/{id}/cancel",
  status="409"
}
```

나쁜 예시는 다음과 같습니다.

```text
http_server_requests_total{
  user_id="984233",
  request_id="7fd8...",
  raw_path="/orders/984233/cancel?from=home"
}
```

앞의 메트릭은 "어떤 API가 어떤 상태 코드로 얼마나 실패하는가"를 즉시 계산할 수 있습니다.
뒤의 메트릭은 결국 개별 요청 추적을 메트릭 저장소에 억지로 밀어 넣은 형태라 비용 대비 판단력이 낮습니다.

## route 템플릿과 bucket 설계가 핵심이다

HTTP 메트릭에서는 원본 path 대신 route 템플릿을 써야 합니다.
`/users/123/orders/456` 대신 `/users/{userId}/orders/{orderId}`처럼 정규화된 라벨을 쓰면 시계열 수를 통제할 수 있습니다.
Spring Boot, Micrometer, OpenTelemetry 계열 계측기들이 route pattern을 지원하는 이유도 여기에 있습니다.

히스토그램도 같은 원리입니다.
버킷을 너무 많이 잡으면 각 버킷이 또 다른 시계열이 되어 비용이 늘어납니다.
대시보드에서 거의 쓰지 않는 초미세 버킷을 많이 두기보다, 실제 SLO 판단에 필요한 구간만 남기는 편이 낫습니다.
예를 들어 API 지연 시간을 본다면 "10ms 단위로 50개 버킷"보다 "50ms, 100ms, 250ms, 500ms, 1s"처럼 운영 기준에 맞는 버킷이 실용적입니다.

## 장애 패턴은 보통 배포 후에 드러난다

카디널리티 문제는 로컬 테스트에서 잘 드러나지 않습니다.
샘플 데이터가 적고, 요청 수가 작고, 파드 개수도 적기 때문입니다.
운영에서는 다음 같은 변경이 갑자기 비용을 키웁니다.

- 신규 라벨에 `tenant_id`를 추가했다.
- 원본 URL 전체를 path 라벨에 그대로 넣었다.
- 배치 작업이 job id를 라벨로 쏟아내기 시작했다.
- 에러 코드를 사람이 읽기 쉬우라고 예외 메시지 자체로 라벨링했다.

이런 실수는 코드상으로는 "관측성 강화"처럼 보이지만, 실제로는 Prometheus의 head series 수를 밀어 올리고 압축 효율을 떨어뜨립니다.
결과적으로 스크레이프 지연, 쿼리 타임아웃, 원격 저장 쓰기 지연이 따라올 수 있습니다.

## 운영에서 확인할 것

카디널리티를 관리할 때는 새 메트릭을 추가할 때만 보는 것이 아니라, 운영 중 추세를 계속 봐야 합니다.

- 활성 시계열 수가 배포 전후로 얼마나 늘었는가
- 특정 잡(job)이나 인스턴스에서 샘플 수가 급증했는가
- 쿼리 응답 시간이 특정 대시보드에서만 느려지는가
- 라벨 값 분포가 예상한 집계 차원만 유지하는가
- 장기 보관 시스템의 저장량과 비용이 갑자기 늘었는가

애플리케이션 리뷰 단계에서는 "이 라벨로 실제로 어떤 질문에 답할 것인가"를 먼저 묻는 편이 좋습니다.
답이 모호하면 대개 로그나 트레이스가 더 맞는 데이터입니다.

## 언제 자세히 쪼개고, 언제 단순화해야 하나

서비스를 운영하다 보면 세부 라벨을 더 넣고 싶은 압박이 늘 있습니다.
하지만 메트릭 설계는 정보량이 아니라 의사결정 효율로 평가해야 합니다.

- 공통 대시보드와 알림 기준에 직접 쓰이는 차원은 라벨로 남긴다.
- 개별 엔터티 추적이 필요한 값은 로그나 트레이스로 보낸다.
- 무한히 증가하는 식별자, 자유 입력 문자열, 예외 전문은 라벨에서 제외한다.
- 히스토그램 버킷은 SLO와 장애 판별에 필요한 구간만 남긴다.

팀이 메트릭을 새로 추가할 때 "추후에 쓸 수도 있으니 일단 넣자"는 판단은 거의 항상 비쌉니다.
카디널리티는 천천히 쌓이다가 한 번에 비용 문제로 돌아오므로, 처음부터 엄격하게 제한하는 편이 낫습니다.

## 정리

Prometheus에서 위험한 것은 메트릭 개수보다 고유한 시계열 개수입니다.
라벨은 집계를 돕는 차원만 남기고, 사용자 ID나 요청 ID 같은 고카디널리티 값은 로그와 트레이스로 보내야 합니다.
원본 path 대신 route 템플릿을 쓰고, 버킷도 운영 판단에 필요한 만큼만 두는 편이 안전합니다.
좋은 메트릭은 더 많은 데이터를 남기는 메트릭이 아니라, 더 적은 비용으로 더 빠른 판단을 가능하게 하는 메트릭입니다.
