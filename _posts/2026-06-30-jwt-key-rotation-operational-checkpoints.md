## 문제 상황

JWT 서명 키를 교체하는 작업은 문서만 보면 단순해 보입니다. 새 키를 만들고, 애플리케이션이 그 키로 서명하게 바꾸면 끝처럼 보이기 때문입니다. 그런데 운영에서는 이 순간부터 이상한 인증 실패가 생기기 쉽습니다. 어떤 인스턴스는 새 키로 토큰을 발급했고, 어떤 API 서버는 아직 예전 공개 키만 캐시에 들고 있어서 검증에 실패하는 식입니다.

더 까다로운 점은, 키 교체가 배포와 달리 "새 버전만 맞추면 끝"이 아니라는 데 있습니다. 이미 발급된 토큰은 `exp`가 끝날 때까지 살아 있고, 검증 서버는 `kid`에 맞는 공개 키를 계속 찾을 수 있어야 합니다. 이 겹침 구간을 설계하지 않으면 로그인은 되는데 다음 요청부터 401이 나는 사고가 발생합니다.

실무에서 중요한 질문은 두 가지입니다. 첫째, 새 키를 언제 공개하고 언제 서명 전환을 할 것인가. 둘째, 예전 키를 언제 제거할 것인가. 키 생성 자체보다 이 순서를 틀리지 않는 편이 훨씬 중요합니다.

## 핵심 개념

JWT는 서명된 토큰이고, 검증 서버는 토큰의 헤더와 클레임만 보는 것이 아니라 "어떤 키로 서명되었는지"까지 같이 판단해야 합니다. RFC 7517은 JWK의 `kid`를 특정 키를 식별하는 값으로 설명합니다. 운영에서는 이 `kid`가 검증 서버가 공개 키를 고르는 기준이 됩니다. 즉, 키를 교체한다는 말은 사실상 "같은 issuer 아래에 여러 공개 키가 일정 기간 공존한다"는 뜻입니다.

RFC 7519는 `exp`가 지난 토큰은 처리하면 안 된다고 설명하고, 작은 clock skew를 고려한 약간의 여유는 둘 수 있다고 말합니다. 이 규칙 때문에 예전 키는 "배포가 끝난 시점"이 아니라 "그 키로 발급된 마지막 토큰의 만료 시각 + 허용 leeway"까지 남겨두는 편이 안전합니다.

또 하나 중요한 점은 알고리즘 검증입니다. RFC 8725는 라이브러리가 허용할 알고리즘 집합을 호출자가 명시할 수 있어야 하고, 실제 암호 연산에 쓰는 알고리즘과 헤더의 `alg`가 일치해야 한다고 권고합니다. 키를 교체할 때 `kid`만 보는 것이 아니라, 이 키가 원래 의도한 알고리즘과 용도인지도 함께 확인해야 한다는 뜻입니다.

## 코드로 보기

아래처럼 서명 키를 여러 개 두고 `kid`로 선택하는 구조를 생각하면 운영 체크포인트가 명확해집니다.

```json
{
  "keys": [
    {
      "kty": "RSA",
      "alg": "RS256",
      "use": "sig",
      "kid": "2026-06-rotate-a"
    },
    {
      "kty": "RSA",
      "alg": "RS256",
      "use": "sig",
      "kid": "2026-06-rotate-b"
    }
  ]
}
```

운영 순서는 대체로 아래가 안전합니다.

1. 새 키를 생성하고 JWKS에 먼저 공개한다.
2. 검증 서버가 새 `kid`를 읽을 수 있는지 확인한다.
3. 발급 서버의 active signing key를 새 `kid`로 바꾼다.
4. 예전 키로 발급된 토큰이 모두 만료될 때까지 예전 공개 키를 유지한다.
5. 만료 구간이 끝난 뒤 예전 키를 제거한다.

Spring 기반 서비스라면 검증 로직에서 최소한 아래와 같은 조건을 고정하는 편이 좋습니다.

```java
if (!allowedAlgorithms.contains(jwt.getHeader().getAlgorithm())) {
    throw new SecurityException("unsupported alg");
}

if (!expectedIssuer.equals(jwt.getIssuer())) {
    throw new SecurityException("invalid iss");
}

if (!jwt.getAudience().contains(expectedAudience)) {
    throw new SecurityException("invalid aud");
}
```

핵심은 "키를 찾았는가"에서 끝나지 않고, `alg`, `iss`, `aud`까지 같이 검증해야 한다는 점입니다.

## 자주 하는 실수

첫 번째 실수는 JWKS를 바꾸자마자 예전 키를 지우는 것입니다. 이 경우 새 토큰은 통과해도, 아직 만료되지 않은 예전 토큰이 갑자기 전부 실패합니다. 모바일 앱이나 배치 작업처럼 토큰을 오래 들고 있는 클라이언트가 있으면 더 크게 터집니다.

두 번째 실수는 공개 키 캐시를 너무 공격적으로 두는 것입니다. 검증 서버가 JWKS를 오래 캐시하면, 새 키가 공개되어도 한동안 `kid not found`가 발생합니다. 그렇다고 요청마다 JWKS를 다시 읽으면 지연과 외부 의존성이 커집니다. 보통은 짧은 TTL 캐시를 두고, `kid`를 못 찾았을 때만 즉시 재조회하는 fallback을 넣는 편이 현실적입니다.

세 번째 실수는 키 교체 작업인데도 인증 실패 로그에 `kid`, `iss`, `aud`, `alg`를 남기지 않는 것입니다. 그러면 401이 났을 때 "키가 없는 건지, issuer가 다른 건지, 알고리즘이 틀린 건지"를 구분하지 못합니다.

## 언제 쓰면 좋은가

키 교체 체계를 꼭 엄격하게 설계해야 하는 경우는 외부 클라이언트가 많거나, 토큰 유효 시간이 길거나, 인스턴스가 여러 대인 환경입니다. 이런 환경에서는 수동 교체나 "배포 끝났으니 지워도 되겠지" 방식이 거의 항상 사고를 만듭니다.

반대로 내부 테스트용 단일 서비스이고 토큰 TTL이 매우 짧다면 구조를 단순하게 가져갈 수 있습니다. 그래도 `kid`를 두고 예전 키를 잠깐 공존시키는 기본 틀은 유지하는 편이 좋습니다. 나중에 서비스가 커질 때 가장 먼저 재활용되는 운영 패턴이기 때문입니다.

실무 판단 기준을 하나만 고르면 이렇습니다. "예전 키를 지우는 시점은 새 키 배포 완료 시점이 아니라, 예전 키로 발급된 토큰의 최장 생존 시간 이후"로 잡는 편이 안전합니다.

## 운영에서 볼 것

- `kid not found` 검증 실패 건수
- `invalid signature`, `invalid issuer`, `invalid audience`처럼 실패 이유별 카운트
- JWKS 재조회 횟수와 실패율
- 로그인 성공 직후 다음 API 호출이 401로 떨어지는 패턴
- 여러 인스턴스 중 특정 인스턴스에서만 실패가 몰리는지 여부

가능하면 인증 실패 로그에 아래 필드는 남기는 편이 좋습니다.

- `kid`
- `alg`
- `iss`
- `aud`
- 검증 서버 인스턴스 ID
- JWKS 캐시 hit/miss 여부

이 정보가 있어야 키 교체 실패를 "보안 사고"가 아니라 "운영 절차 문제"로 빠르게 좁힐 수 있습니다.

## 정리

JWT 키 교체는 새 키를 만드는 작업보다, 예전 키와 새 키가 함께 살아 있는 기간을 관리하는 작업에 가깝습니다. `kid` 선택, `exp` 만료 구간, 알고리즘 검증, JWKS 캐시 전략을 같이 설계해야 장애가 줄어듭니다. 운영에서는 키 제거 시점을 배포 완료 기준으로 잡지 말고, 마지막 예전 토큰의 만료 시각 기준으로 잡는 편이 안전합니다.
