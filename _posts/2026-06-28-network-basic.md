백엔드 개발자는 코드를 작성하는 것만큼이나 요청이 서버까지 도착하는 흐름을 이해해야 합니다.
API가 느리거나 연결이 실패할 때 원인을 좁히려면 네트워크의 기본 단어들이 익숙해야 합니다.

## 요청은 어디를 지나갈까

브라우저에서 `https://example.com`에 접속하면 대략 이런 순서가 일어납니다.

1. 도메인을 IP 주소로 바꾸기 위해 DNS 조회를 합니다.
2. 클라이언트는 목적지 IP와 포트로 연결을 시도합니다.
3. TCP 연결이 만들어집니다.
4. HTTPS라면 TLS 핸드셰이크로 암호화 통신 준비를 합니다.
5. HTTP 요청이 서버로 전달됩니다.
6. 서버는 응답을 만들고 다시 클라이언트로 보냅니다.

이 흐름을 알면 "서버 코드가 느린 것인지", "DNS나 네트워크가 문제인지", "TLS나 프록시 설정이 문제인지"를 나누어 볼 수 있습니다.

## TCP와 UDP

TCP는 연결 지향 프로토콜입니다. 데이터를 순서대로, 손실 없이 보내는 것을 목표로 합니다.
HTTP/1.1, HTTP/2는 TCP 위에서 동작합니다.

UDP는 연결을 만들지 않고 데이터를 보냅니다. 손실 가능성은 있지만 빠르고 단순합니다.
DNS 조회, 스트리밍, 게임 서버처럼 지연 시간이 중요한 상황에서 자주 사용됩니다.

## 포트는 애플리케이션의 문이다

IP가 서버의 주소라면 포트는 서버 안에서 어떤 프로그램으로 들어갈지를 나타냅니다.

- `80`: HTTP
- `443`: HTTPS
- `3306`: MySQL
- `6379`: Redis
- `8080`: 개발 서버에서 자주 사용하는 HTTP 포트

서버가 실행 중인데 접속이 안 된다면 애플리케이션, 방화벽, 보안 그룹, 포트 바인딩을 함께 확인해야 합니다.

## 백엔드 개발자가 기억할 점

장애를 볼 때는 계층을 나누어 확인하는 습관이 중요합니다.

- DNS가 올바른 IP를 가리키는가
- 서버 포트가 열려 있는가
- 프록시나 로드밸런서 설정이 맞는가
- 애플리케이션 로그에 요청이 도착했는가
- 응답 시간이 어느 구간에서 길어지는가

네트워크는 어렵지만, 요청의 경로를 머릿속에 그릴 수 있으면 문제를 훨씬 차분하게 추적할 수 있습니다.
