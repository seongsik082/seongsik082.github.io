백엔드 테스트 코드는 버그를 줄이기 위한 도구이면서 동시에 코드의 의도를 설명하는 문서입니다.
테스트가 있으면 기능을 수정할 때 기존 동작이 깨졌는지 더 빨리 알 수 있습니다.

## 테스트의 목적

테스트를 작성할 때 가장 먼저 생각할 것은 "무엇을 보장하고 싶은가"입니다.
모든 줄을 테스트하려고 하면 금방 지치고, 중요한 시나리오가 빠질 수 있습니다.

백엔드에서는 보통 다음을 우선 확인합니다.

- 입력값 검증이 제대로 되는가
- 비즈니스 규칙이 지켜지는가
- 예외 상황에서 기대한 응답이 나오는가
- DB 저장과 조회가 의도대로 동작하는가
- 외부 API 실패를 처리할 수 있는가

## 단위 테스트

단위 테스트는 작은 코드 조각을 빠르게 검증합니다.
도메인 객체, 계산 로직, 검증 로직처럼 외부 의존성이 적은 코드를 테스트하기 좋습니다.

```java
@Test
void 이름이_비어있으면_회원가입할_수_없다() {
    assertThatThrownBy(() -> new User(""))
        .isInstanceOf(IllegalArgumentException.class);
}
```

단위 테스트는 빠르게 실행되어야 합니다.
그래야 개발 중에 자주 돌릴 수 있습니다.

## 통합 테스트

통합 테스트는 여러 계층이 함께 동작하는지 확인합니다.
예를 들어 Controller, Service, Repository, DB가 연결된 흐름을 검증할 수 있습니다.

통합 테스트는 단위 테스트보다 느리지만, 실제 서비스 흐름을 더 잘 확인할 수 있습니다.
API 요청과 응답을 검증할 때 특히 유용합니다.

## 테스트 이름

테스트 이름은 실패했을 때 무엇이 깨졌는지 알려줘야 합니다.

좋은 테스트 이름은 보통 조건과 기대 결과를 함께 담습니다.

```text
이미_가입된_이메일이면_회원가입에_실패한다
게시글이_없으면_404를_반환한다
권한이_없는_사용자는_게시글을_삭제할_수_없다
```

## 정리

테스트는 완벽한 안전망이 아니라 변경 비용을 줄이는 장치입니다.
처음부터 모든 테스트를 작성하려고 하기보다, 중요한 규칙과 자주 깨지는 흐름부터 테스트하는 것이 좋습니다.

백엔드 개발자는 기능 구현만큼이나 "이 기능이 계속 올바르게 동작한다는 근거"를 남기는 습관이 필요합니다.
