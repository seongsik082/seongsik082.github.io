백엔드 개발에서 API는 클라이언트와 서버 사이의 약속입니다.
좋은 API는 기능만 동작하는 것이 아니라, 요청과 응답의 의도가 명확하게 드러납니다.

## HTTP 메서드

HTTP 메서드는 서버에 어떤 작업을 요청하는지 표현합니다.

- `GET`: 리소스를 조회합니다.
- `POST`: 새 리소스를 생성하거나 명령을 실행합니다.
- `PUT`: 리소스 전체를 교체합니다.
- `PATCH`: 리소스 일부를 수정합니다.
- `DELETE`: 리소스를 삭제합니다.

예를 들어 게시글 API를 만든다면 다음처럼 표현할 수 있습니다.

```text
GET    /posts
GET    /posts/{id}
POST   /posts
PATCH  /posts/{id}
DELETE /posts/{id}
```

URI에는 동사보다 명사를 쓰고, 실제 행위는 HTTP 메서드로 드러내는 것이 일반적입니다.

## 상태 코드

상태 코드는 요청 결과를 숫자로 표현합니다.

- `200 OK`: 요청 성공
- `201 Created`: 생성 성공
- `400 Bad Request`: 잘못된 요청
- `401 Unauthorized`: 인증 필요
- `403 Forbidden`: 권한 없음
- `404 Not Found`: 리소스 없음
- `500 Internal Server Error`: 서버 내부 오류

상태 코드를 일관되게 쓰면 프론트엔드와 협업하기 쉬워지고, 장애 상황에서도 원인을 더 빨리 좁힐 수 있습니다.

## 응답 바디

API 응답은 예측 가능해야 합니다.
성공 응답과 실패 응답의 형식을 어느 정도 통일해두면 클라이언트가 처리하기 쉽습니다.

```json
{
  "code": "POST_NOT_FOUND",
  "message": "게시글을 찾을 수 없습니다."
}
```

에러 메시지는 사용자에게 보여줄 문장과 개발자가 추적할 코드를 함께 고려하는 것이 좋습니다.

## 백엔드 API 설계 체크리스트

- URI가 리소스 중심으로 읽히는가
- HTTP 메서드가 의도에 맞는가
- 성공과 실패 상태 코드가 일관적인가
- 에러 응답 형식이 통일되어 있는가
- 인증 실패와 권한 실패를 구분하는가
- 페이징, 정렬, 검색 조건이 명확한가

API 설계는 작은 약속의 누적입니다. 처음부터 완벽할 필요는 없지만, 일관성을 지키면 유지보수가 훨씬 쉬워집니다.
