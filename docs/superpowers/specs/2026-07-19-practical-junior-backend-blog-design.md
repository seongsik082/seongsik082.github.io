# 실무형 주니어 백엔드 블로그 전환 설계

## 목표

기술 블로그를 매일 공식 문서를 요약하는 채널에서, 주니어 백엔드 개발자가 실제 설계와 운영 판단을 따라 할 수 있는 기록으로 전환한다. 독자는 정의보다 “무엇을 먼저 확인하고, 어떤 선택을 피해야 하며, 왜 그렇게 결정하는가”를 얻어야 한다.

## 독자와 문체

- 주 독자는 Java·Spring·데이터베이스를 학습한 주니어 백엔드 개발자다.
- 처음 등장하는 용어는 한 문장으로 뜻을 설명하되, 기초 개념 나열을 본문 중심으로 삼지 않는다.
- 차분한 설명체를 쓰고, 모든 글에 같은 도입·체크리스트·요약 문구를 반복하지 않는다.
- 실제 운영 경험이 없는 내용을 운영 장애처럼 서술하지 않는다. 사례의 상태를 `구현됨`, `재현 실험`, `설계 시나리오` 중 하나로 명시한다.

## 게시 주기와 주제 구성

- 기존의 매일 3개 게시를 중단하고, 매주 월요일과 목요일 오전 9시(Asia/Seoul)에 한 편씩 게시한다.
- 한 편의 한국어 본문은 참고 링크를 제외하고 5,000~7,000자를 목표로 한다. 근거가 부족하면 분량을 채우지 않고 게시를 보류한다.
- 최근 4주 글의 주제는 다음 비중으로 관리한다.
  - 40%: `JD Proof`의 공고·요구사항·근거·매칭 점수·액션 과제 설계
  - 40%: `AgentOps Board KR`의 권한·승인·감사 로그·실행 재현·평가 설계
  - 20%: 위 프로젝트에서 파생한 재현 가능한 데이터베이스·API·보안·운영 실험
- 무작위 카테고리 분산은 보조 기준일 뿐이다. 현재 프로젝트의 실제 결정이나 검증 가능한 질문이 없으면 새 글을 만들지 않는다.

## 글의 근거 등급

각 글의 도입부에는 사례 상태와 근거를 한 문장으로 표시한다. 공개할 수 없는 값, 사용자 정보, 토큰, 원문 이력은 넣지 않는다.

| 등급 | 사용할 수 있는 근거 | 표현 규칙 |
| --- | --- | --- |
| 구현됨 | 저장소 코드, API 명세, DB migration, 테스트 결과 | 관찰한 동작과 한계를 구분해 적는다. |
| 재현 실험 | 실행 명령, 입력값, 환경, 로그·쿼리·측정값 | 환경과 재현 절차를 밝히고, 결과를 일반화하지 않는다. |
| 설계 시나리오 | 프로젝트 요구사항, 데이터 모델, 공식 문서 | 아직 운영 결과가 없음을 밝히고, 선택지와 검증 계획을 함께 적는다. |

## 글 형식

모든 글에 같은 제목 순서를 강제하지 않는다. 다만 아래 요소 중 필수 요소를 빠뜨리지 않는다.

1. 사용자가 실제로 겪을 수 있는 요청 흐름 또는 설계 제약을 제시한다.
2. 프로젝트 API, 테이블, 설정, SQL, 로그, 테스트 결과 중 최소 두 가지 구체적 산출물을 제공한다.
3. 최소 두 가지 대안을 같은 제약 조건에서 비교하고, 선택하지 않은 이유를 설명한다.
4. 주니어가 실행할 수 있는 검증 절차와 실패했을 때의 첫 확인 지점을 적는다.
5. 적용하지 말아야 할 조건 또는 비용을 하나 이상 분명히 적는다.
6. 주장에 직접 관련된 공식 문서·RFC·프로젝트 공식 문서를 링크한다.

권장 형식은 세 가지다.

- **설계 결정 기록**: 요구사항 → 선택지 비교 → API·DB 경계 → 실패 시나리오 → 검증 계획
- **재현 실험 기록**: 증상 → 환경과 입력 → 관찰 자료 → 원인 가설 검증 → 수정과 재측정
- **운영 흐름 해설**: 실제 요청 흐름 → 위험 지점 → 설정·로그 해석 → 대응 기준

## 첫 두 주제 흐름

첫 번째 글은 `JD Proof`에서 LLM 추출 결과를 `Requirement`와 `Evidence`로 분리하는 이유를 다룬다. 같은 공고 문장이라도 근거 링크 없이 매칭 점수를 확정하면 안 되는 이유, 데이터 모델, 검토 가능한 API 응답, 사람이 수정하는 경계를 설명한다. 사례 상태는 설계 시나리오다.

두 번째 글은 `AgentOps Board KR`에서 승인 전후 실행을 하나의 감사 가능한 흐름으로 만들기 위한 `Run`, `Approval`, `StepLog`의 경계를 다룬다. 승인 중복 요청, 재실행, 민감정보 원문 노출을 어떻게 피할지와 idempotency·감사 로그의 trade-off를 설명한다. 사례 상태는 설계 시나리오다.

이후에는 구현 결과가 생긴 기능을 우선하고, 구현 전에는 설계 시나리오임을 유지한다. 성능 개선 수치, 장애 빈도, 사용량 수치는 직접 측정하지 않은 한 쓰지 않는다.

## 2026-07-19 게시물 처리

기존 세 글은 URL과 게시일을 유지하되, 제목·excerpt·본문을 전면 개정한다. 공식 문서 요약을 늘리는 방식이 아니라 이 블로그와 현재 프로젝트의 판단을 연결한다.

- `oauth2-pkce-code-verifier-server-validation`: JD Proof에서 외부 GitHub 연동을 설계할 때 public client와 PKCE 검증을 어디에 두는지, callback·state·token 저장 경계를 설계 시나리오로 설명한다.
- `github-actions-concurrency-deployment-order`: GitHub Pages 직접 배포와 서비스 배포를 구분하고, AgentOps Board KR의 CI/CD에서 deploy lock·migration·취소 정책을 어떻게 설계할지를 설계 시나리오로 설명한다.
- `redis-rdb-aof-recovery-data-loss-window`: JD Proof의 작업 상태와 캐시를 분리하고, Redis를 원본 데이터로 쓰지 않는 이유와 RPO·복구 리허설을 재현 실험 계획으로 설명한다.

세 글 모두 가상의 프로덕션 장애를 실제 사건처럼 쓰지 않는다. 구현·실험하지 않은 결과는 “검증할 항목”으로 남긴다.

## 자동화 품질 게이트

자동화는 게시 전에 아래 조건을 모두 통과해야 한다.

- 게시 주기가 월·목 09:00 KST인지 확인한다.
- 주제가 최근 글과 다른지뿐 아니라 프로젝트의 기능·결정·실험과 연결되는지 확인한다.
- 본문 5,000자 이상, 7,000자 이하인지 확인한다.
- 사례 상태와 근거 등급을 명시했는지 확인한다.
- 구체적 산출물 두 가지, 비교한 대안 두 가지, 적용하지 않을 조건 한 가지가 있는지 사람이 검토한다.
- 공식 출처 링크와 본문 설명이 일치하는지 확인한다.
- 기존의 Jekyll front matter, 날짜, 공백, 중복 검사와 Git 검증을 수행한다.

품질 게이트를 충족하지 못하면 해당 회차를 게시하지 않는다. 빈도를 지키기 위해 문장 반복이나 근거 없는 수치를 추가하지 않는다.

## 범위와 비범위

이번 변경은 `_posts/`의 세 게시물, 글 작성 규칙, 자동화 지침과 검증 절차에 한정한다. 사이트 레이아웃, 기존 사용자 변경 파일, 포트폴리오 원문, 채용공고 수집기, Jekyll 빌드는 변경하지 않는다.

## 완료 기준

- 세 기존 게시물이 위 근거 등급과 글 형식을 만족하도록 전면 개정된다.
- 자동화가 월·목 오전 9시 KST에 한 편만 작성하도록 바뀐다.
- 다음 자동 작성부터 프로젝트 근거·분량·품질 게이트를 통과하지 못한 글은 게시하지 않는다.
- 새 글과 수정 글은 YAML front matter, 미래 날짜, 공백, 링크, Git 상태 검증을 통과한다.
